    Overview¶

    This API Reference consolidates comprehensive information about the Maily platform's REST API, which enables programmatic access to email marketing functionality. This document serves as the single source of truth for developers integrating with the Maily platform.

    The Maily API is organized around REST principles. It uses standard HTTP verbs, returns JSON responses, includes predictable resource-oriented URLs, and uses HTTP status codes to indicate errors. All API access is over HTTPS, and all data is sent and received as JSON.

    Key features of the Maily API:

    • Comprehensive Coverage: Access to all Maily platform capabilities
    • Consistent Design: Predictable URL patterns and response structures
    • Secure: TLS encryption and multiple authentication options
    • Versioned: Backward compatibility with clear deprecation policies
    • Developer-Friendly: Detailed documentation and SDKs

    Authentication¶

    The Maily API supports three authentication methods:

    API Keys¶

    API keys provide simple token-based authentication. Include your API key in the request header:

    GET /api/v1/campaigns HTTP/1.1
Host: api.maily.io
X-API-Key: your_api_key_here

    API key management: - Generate keys in the account settings page - Each key has configurable permissions - Always keep keys secure and never expose them in client-side code - Rotate keys regularly for security

    OAuth 2.0¶

    OAuth 2.0 is supported for integrations requiring user authorization:

    1. Register your application in the Maily developer portal
    2. Redirect users to the authorization URL:
    https://app.maily.io/oauth/authorize?
  client_id=YOUR_CLIENT_ID&
  redirect_uri=YOUR_REDIRECT_URI&
  response_type=code&
  scope=campaigns:read subscribers:write
    1. Exchange the authorization code for an access token:
    POST /oauth/token HTTP/1.1
Host: api.maily.io
Content-Type: application/x-www-form-urlencoded

grant_type=authorization_code&
code=AUTHORIZATION_CODE&
client_id=YOUR_CLIENT_ID&
client_secret=YOUR_CLIENT_SECRET&
redirect_uri=YOUR_REDIRECT_URI
    1. Use the access token in your API requests:
    GET /api/v1/campaigns HTTP/1.1
Host: api.maily.io
Authorization: Bearer YOUR_ACCESS_TOKEN

    JWT Authentication¶

    For server-to-server integrations, JWT authentication is recommended:

    1. Generate a JWT using your API secret:
    const jwt = require('jsonwebtoken');
const payload = {
  iss: 'your_app_id',
  exp: Math.floor(Date.now() / 1000) + (60 * 60), // 1 hour expiration
  iat: Math.floor(Date.now() / 1000)
};

const token = jwt.sign(payload, 'your_api_secret');
    1. Include the JWT in your API requests:
    GET /api/v1/campaigns HTTP/1.1
Host: api.maily.io
Authorization: Bearer YOUR_JWT_TOKEN

    Request Guidelines¶

    Base URL¶

    All API requests should be made to the following base URL:

    https://api.maily.io/api/v1/

    For region-specific deployments:

    https://{region}.api.maily.io/api/v1/

    Where {region} is one of: - us - United States (default) - eu - European Union - ap - Asia Pacific - ca - Canada

    Headers¶

    The following headers should be included with every request:

    Header Required Description
    Content-Type Yes Set to application/json for request bodies
    Accept Yes Set to application/json
    X-API-Key or Authorization Yes Authentication credentials
    X-Idempotency-Key No Unique key for idempotent requests
    X-Request-ID No Client-generated request identifier for tracing

    Example headers:

    Content-Type: application/json
Accept: application/json
X-API-Key: your_api_key_here
X-Idempotency-Key: 123e4567-e89b-12d3-a456-426614174000
X-Request-ID: req_1234567890

    Request Format¶

    For POST, PUT, and PATCH requests, the request body should be valid JSON:

    {
  "name": "Black Friday Campaign",
  "subject": "Don't Miss Our Biggest Sale!",
  "content": {
    "html": "<html><body><h1>Sale Now!</h1></body></html>",
    "text": "Sale Now!"
  },
  "list_id": "list_1a2b3c4d5e",
  "schedule": {
    "send_at": "2025-11-27T08:00:00Z"
  }
}

    For GET requests, parameters should be included in the query string:

    /api/v1/campaigns?status=draft&limit=10&page=2

    Rate Limits¶

    The Maily API implements rate limiting to ensure fair usage and platform stability:

    Plan Rate Limit Burst Limit
    Free 60 requests/minute 100 requests/minute
    Standard 180 requests/minute 300 requests/minute
    Premium 600 requests/minute 1000 requests/minute
    Enterprise Custom Custom

    Rate limit headers are included in all responses:

    X-RateLimit-Limit: 180
X-RateLimit-Remaining: 179
X-RateLimit-Reset: 1614556800

    When rate limits are exceeded, the API returns a 429 Too Many Requests status code with a Retry-After header indicating when to retry.

    Response Guidelines¶

    Status Codes¶

    The Maily API uses conventional HTTP status codes to indicate the success or failure of requests:

    Code Description
    200 OK - The request was successful
    201 Created - The resource was successfully created
    204 No Content - The request was successful, but no content is returned
    400 Bad Request - The request contains invalid parameters
    401 Unauthorized - Authentication credentials are missing or invalid
    403 Forbidden - The authenticated user doesn't have permission
    404 Not Found - The requested resource doesn't exist
    422 Unprocessable Entity - The request is well-formed but cannot be processed
    429 Too Many Requests - Rate limit exceeded
    500 Internal Server Error - An error occurred on the server
    503 Service Unavailable - The service is temporarily unavailable

    Response Format¶

    All API responses are returned in JSON format. A typical successful response follows this structure:

    {
  "data": {
    "id": "camp_1a2b3c4d5e",
    "name": "Black Friday Campaign",
    "subject": "Don't Miss Our Biggest Sale!",
    "status": "draft",
    "created_at": "2025-03-01T12:00:00Z",
    "updated_at": "2025-03-01T12:00:00Z"
  },
  "meta": {
    "request_id": "req_1234567890"
  }
}

    For collection endpoints, the response includes an array of objects and pagination information:

    {
  "data": [
    {
      "id": "camp_1a2b3c4d5e",
      "name": "Black Friday Campaign",
      "status": "draft"
    },
    {
      "id": "camp_2f3g4h5i6j",
      "name": "Welcome Series",
      "status": "active"
    }
  ],
  "meta": {
    "pagination": {
      "total": 42,
      "count": 2,
      "per_page": 10,
      "current_page": 1,
      "total_pages": 5,
      "links": {
        "next": "https://api.maily.io/api/v1/campaigns?page=2"
      }
    }
  }
}

    Error responses follow a standard format:

    {
  "error": {
    "code": "invalid_parameter",
    "message": "The provided campaign name is too long",
    "details": [
      {
        "field": "name",
        "message": "Must be less than 100 characters"
      }
    ]
  },
  "meta": {
    "request_id": "req_1234567890"
  }
}

    Pagination¶

    Collection endpoints support pagination through the following query parameters:

    Parameter Default Description
    page 1 Page number to retrieve
    per_page 10 Number of items per page (max 100)
    sort created_at Field to sort by
    order desc Sort order (asc or desc)

    The response includes pagination metadata and links for navigating between pages:

    "meta": {
  "pagination": {
    "total": 42,
    "count": 10,
    "per_page": 10,
    "current_page": 1,
    "total_pages": 5,
    "links": {
      "next": "https://api.maily.io/api/v1/campaigns?page=2",
      "last": "https://api.maily.io/api/v1/campaigns?page=5"
    }
  }
}

    Error Handling¶

    The API uses standard HTTP status codes to indicate the result of requests. All error responses include a consistent structure:

    {
  "error": {
    "code": "error_code",
    "message": "Human-readable error message",
    "details": [
      {
        "field": "affected_field",
        "message": "Specific error for this field"
      }
    ]
  },
  "meta": {
    "request_id": "req_1234567890"
  }
}

    Common error codes include:

    Error Code Description
    authentication_required No authentication credentials provided
    invalid_credentials The provided credentials are invalid
    insufficient_permissions The authenticated user lacks required permissions
    resource_not_found The requested resource doesn't exist
    validation_error The request contains invalid data
    rate_limit_exceeded Too many requests have been made in a given time period
    server_error An unexpected error occurred on the server

    POST /campaigns Create a new campaign

    Security¶

    Webhook payloads are signed using HMAC-SHA256. The signature is included in the X-Maily-Signature HTTP header.

    To verify the signature:

    1. Get the webhook secret from your webhook settings in the Maily dashboard
    2. Create an HMAC-SHA256 signature using your webhook secret and the raw request body
    3. Compare the computed signature with the value in the X-Maily-Signature header

    Example verification in Node.js:

    const crypto = require('crypto');

function verifyWebhookSignature(payload, signature, secret) {
  const computedSignature = crypto
    .createHmac('sha256', secret)
    .update(payload)
    .digest('hex');

  return crypto.timingSafeEqual(
    Buffer.from(computedSignature),
    Buffer.from(signature)
  );
}

// In your webhook handler
app.post('/webhooks/maily', (req, res) => {
  const signature = req.headers['x-maily-signature'];
  const payload = JSON.stringify(req.body);
  const secret = 'whsec_your_webhook_secret';

  if (!verifyWebhookSignature(payload, signature, secret)) {
    return res.status(401).send('Invalid signature');
  }

  // Process the webhook
  const event = req.body;
  console.log(`Received event: ${event.type}`);

  res.status(200).send('Webhook received');
});

    Deprecation Policy¶

    Maily follows these API lifecycle and deprecation principles:

    1. Version Support: Major API versions are supported for a minimum of 24 months after a new version is released
    2. Deprecation Notice: Features or endpoints to be deprecated will be announced at least 6 months in advance
    3. Backwards Compatibility: We strive to maintain backwards compatibility within a major version
    4. Communication: Deprecation notices are provided via:
    5. Email to account administrators
    6. Notices in the API documentation
    7. HTTP response headers for affected endpoints
