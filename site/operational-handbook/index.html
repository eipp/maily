    Overview¶

    This Operational Handbook serves as the comprehensive guide for operating and maintaining the Maily platform in production environments. It consolidates previously scattered operational documentation into a single source of truth for the operations team, DevOps engineers, and other stakeholders responsible for the platform's reliability, security, and performance.

    The handbook follows these key operational principles:

    1. Automation First: Automate repeated tasks wherever possible
    2. Infrastructure as Code: Manage all infrastructure through code
    3. Observability: Maintain comprehensive visibility into system behavior
    4. Security by Design: Embed security practices into all operations
    5. Continuous Improvement: Regularly review and enhance operational procedures

    Environment Management¶

    Environment Types¶

    Maily uses the following environment types:

    Environment Purpose Access Data
    Development Feature development All developers Anonymized sample data
    Staging Pre-release testing Engineering and QA Anonymized production clone
    Production Live customer instances Limited operator access Customer data
    DR Disaster recovery Automated systems and emergency access Replica of production

    Configuration Management¶

    All environment configuration follows these principles:

    1. Environment Variables: All configuration is passed via environment variables
    2. Secret Management: Secrets are stored in AWS Secrets Manager
    3. Infrastructure as Code: Environment configuration is defined in Terraform
    4. Immutability: Configuration changes trigger new deployments instead of in-place changes

    Configuration values are managed in our hierarchical configuration system:

    config/
├── base.json            # Base configuration
├── development.json     # Development overrides
├── staging.json         # Staging overrides
└── production.json      # Production overrides

    Sensitive values are never committed to the repository and are instead referenced from the secrets manager:

    // Example configuration entry
{
  "database": {
    "host": "${PGHOST}",
    "port": "${PGPORT}",
    "username": "${PGUSER}",
    "password": "${SECRET:PGPASSWORD}"
  }
}

    Access Control¶

    Environment access follows the principle of least privilege:

    Role Development Staging Production DR
    Developer Full Read-only None None
    DevOps Full Full Limited Emergency
    SRE Full Full Full Full
    Support None Read-only Read-only None

    Production access requires: 1. Multi-factor authentication 2. Just-in-time access requests via PagerDuty 3. Session recording for audit purposes 4. Automatic access expiration after 4 hours

    Deployment¶

    Release Process¶

    Maily follows a continuous delivery model with the following release cadence:

    1. Feature Releases: Bi-weekly on Tuesdays
    2. Patch Releases: As needed for critical fixes
    3. Major Releases: Quarterly with advance notice to customers

    The release process:

    1. Code is merged to main after passing all CI checks
    2. Release candidate is created and deployed to staging
    3. Automated and manual testing is performed on staging
    4. Release is scheduled and announced
    5. Progressive deployment to production regions
    6. Post-deployment verification
    7. Release retrospective

    Deployment Pipeline¶

    Deployments are fully automated via our CI/CD pipeline:

    graph TD
    A[Code Merged to Main] --> B[Build and Test]
    B --> C[Create Release Artifact]
    C --> D[Deploy to Staging]
    D --> E[Integration Tests]
    E --> F[Manual Approval]
    F --> G[Progressive Production Deployment]
    G --> H[Post-deployment Verification]
    H --> I[Release Complete]

    Production deployments use a progressive rollout strategy: 1. Deploy to 5% of edge nodes 2. Monitor for 15 minutes 3. If metrics are stable, deploy to 25% 4. Monitor for another 15 minutes 5. If stable, deploy to remaining nodes 6. If issues detected, automatic rollback

    Rollback Procedures¶

    In case of deployment issues, use the following rollback procedures:

    1. Automated Rollback: The deployment pipeline automatically rolls back if error rates or latency exceed thresholds
    2. Manual Rollback: Run the rollback command: 
      ops deploy rollback --env=production --service=<service_name>
    3. Database Rollback: For migrations, follow the database rollback procedure in the Database Operations section

    Feature Flags¶

    Maily uses feature flags to safely deploy changes:

    1. All significant features are behind feature flags
    2. Flags can be toggled per environment, per tenant, or per user
    3. Emergency kill switches exist for all core features
    4. Feature flags are managed through the admin portal or CLI: 
      ops flag enable --env=production --flag=new-email-editor --percentage=10

    Monitoring¶

    System Metrics¶

    Core infrastructure metrics are collected via Datadog agents:

    Metric Category Key Metrics Alert Thresholds
    Compute CPU, Memory, Disk, Network 80% utilization (warning), 90% (critical)
    Database Connections, Query performance, Locks 100 connections, 1s query time, lock waits > 10s
    Cache Hit rate, Memory, Evictions <80% hit rate, >90% memory, any evictions
    Message Queue Queue depth, Processing time >1000 messages, >30s processing
    CDN Cache hit ratio, Bandwidth, Error rate <90% hit rate, >80% capacity, >0.1% error

    Application Metrics¶

    Application-specific metrics are collected via custom instrumentation:

    Metric Description Alert Threshold
    Request Rate Requests per second Sustained 50% change from baseline
    Response Time Time to process requests p95 > 500ms
    Error Rate Percentage of 5xx responses >0.5%
    Active Users Concurrent users N/A (tracking only)
    Email Send Rate Emails sent per minute >5000/minute (capacity planning)
    AI Processing Time Time for AI operations p95 > 2s

    Business Metrics¶

    Key business metrics are tracked for operational insight:

    1. Active users (daily, weekly, monthly)
    2. Email delivery rate and engagement
    3. Campaign creation and scheduling
    4. Subscriber growth and churn
    5. Premium feature usage

    Alerting Strategy¶

    Our alerting follows these principles:

    1. Actionable: Alerts should require human intervention
    2. Clear Ownership: Each alert has a defined owner
    3. Context: Alerts include sufficient context to begin troubleshooting
    4. Prioritized: Alerts are categorized by severity and impact

    Alert routing is configured in PagerDuty:

    Priority Description Notification Response SLA
    P1 Service outage or data loss Call + SMS + Email 15 minutes
    P2 Degraded performance or partial outage SMS + Email 30 minutes
    P3 Non-critical component issue Email 4 hours
    P4 Warning or potential issue Email Next business day

    Dashboards¶

    Operational dashboards are maintained in Datadog and Grafana:

    1. Service Overview: High-level health of all services
    2. User Experience: Performance from user perspective
    3. Infrastructure: Detailed system metrics
    4. Business: Key business metrics
    5. Security: Security-related metrics and events

    Dashboard access is provided to: - Engineering team (all dashboards) - Product team (User Experience and Business dashboards) - Executive team (Business dashboard)

    Incident Management¶

    Severity Levels¶

    Incidents are classified by severity level:

    Level Impact Example Response
    SEV-1 Critical service outage Production is down All-hands emergency
    SEV-2 Major functionality broken Emails not sending Immediate incident response
    SEV-3 Minor functionality affected Reporting delays Scheduled fix
    SEV-4 Cosmetic issues UI alignment problems Backlog item

    On-Call Procedures¶

    The on-call rotation follows a weekly schedule:

    1. Primary on-call: Front-line response to all alerts
    2. Secondary on-call: Backup for primary and escalation point
    3. Engineering manager: Escalation for major incidents

    On-call handoff process: 1. Daily handover meetings at 4pm 2. Written handover notes documenting ongoing issues 3. Acknowledgment of handover by incoming on-call

    On-call tools: - PagerDuty for alerting and scheduling - Slack for communication - Zoom for incident calls - Incident response documentation in Notion

    Incident Response¶

    The incident response process follows these steps:

    1. Acknowledge: Acknowledge the alert and begin investigation
    2. Assess: Determine the scope and impact of the incident
    3. Communicate: Post initial status to the incident channel
    4. Mitigate: Implement immediate mitigation steps
    5. Resolve: Fix the underlying issue
    6. Document: Record details for the postmortem

    For SEV-1 and SEV-2 incidents: 1. Create an incident channel in Slack (#incident-{date}-{short-description}) 2. Designate an Incident Commander (IC) 3. Designate a Communications Lead 4. Update status page within 15 minutes 5. Provide updates at least every 30 minutes

    Postmortem Process¶

    After each SEV-1 or SEV-2 incident:

    1. Complete the postmortem template within 2 business days
    2. Include timeline, root cause, and action items
    3. Present in the weekly engineering meeting
    4. Track action items to completion

    The postmortem should be blameless and focus on: - What happened - Why it happened - How to prevent similar incidents - How to improve response

    Database Operations¶

    Backup and Recovery¶

    Database backup strategy:

    Database Backup Type Frequency Retention
    Primary (PostgreSQL) Full backup Daily 30 days
    Primary (PostgreSQL) Incremental Hourly 7 days
    Primary (PostgreSQL) Write-ahead logs Continuous 7 days
    Redis RDB snapshot Every 6 hours 7 days
    Document store Full backup Daily 30 days

    Backup testing: 1. Monthly restoration test to staging environment 2. Quarterly disaster recovery simulation

    Performance Tuning¶

    Database performance monitoring:

    1. Query performance is tracked via pgStatStatements
    2. Slow query log threshold is set to 100ms
    3. Weekly review of slow queries
    4. Monthly index optimization

    Production database guidelines: - Max connections: 500 - Statement timeout: 30s - Idle in transaction timeout: 5m - Maintenance window: Sundays 2-4 AM EST

    Migrations in Production¶

    Guidelines for production database migrations:

    1. All migrations must be backward compatible
    2. Large migrations should be split into smaller steps
    3. Add before remove (add columns/tables before removing them)
    4. Include a rollback plan for each migration
    5. Test migrations on a production clone before deployment
    6. Schedule high-risk migrations during maintenance windows

    Execution process: 

    # Pre-migration check
ops db check-migration --env=production

# Apply migration
ops db migrate --env=production

# Verify migration
ops db verify-migration --env=production

    Data Retention¶

    Data retention policies:

    Data Type Retention Period Archival Strategy
    Customer data Indefinite Active database
    Email analytics 2 years Database → S3
    System logs 90 days CloudWatch → S3
    Audit logs 7 years Database → S3
    Session data 30 days Redis → purged

    Data purging jobs run weekly during low-traffic periods.

    Security Operations¶

    Vulnerability Management¶

    Security scanning process:

    1. Dependency Scanning: Automated in CI pipeline
    2. Container Scanning: Images scanned before deployment
    3. Infrastructure Scanning: Weekly cloud configuration audit
    4. Penetration Testing: Quarterly by external vendor
    5. Bug Bounty Program: Continuous via HackerOne

    Vulnerability response timeline:

    Severity Response Time Remediation SLA
    Critical 24 hours 7 days
    High 48 hours 14 days
    Medium 1 week 30 days
    Low 2 weeks 90 days

    Access Reviews¶

    Access control reviews:

    1. All production access is reviewed monthly
    2. Cloud IAM permissions reviewed quarterly
    3. Database access reviewed quarterly
    4. Service account permissions reviewed quarterly

    User access lifecycle: - Onboarding: Access granted based on role template - Role changes: Access adjusted within 1 business day - Offboarding: Access revoked immediately

    Security Monitoring¶

    Security monitoring systems:

    1. SIEM: Datadog and CloudWatch Logs
    2. WAF: AWS WAF with custom rules
    3. Network Monitoring: VPC Flow Logs
    4. Endpoint Detection: CrowdStrike
    5. Identity Monitoring: Unusual login detection

    Key security alerts: - Multiple failed login attempts - Unusual access patterns - Configuration changes outside deployment windows - Sensitive data access - Privilege escalation

    Incident Response¶

    Security incident response process:

    1. Containment: Isolate affected systems
    2. Investigation: Determine scope and impact
    3. Eradication: Remove threat and vulnerabilities
    4. Recovery: Restore systems to normal operation
    5. Lessons Learned: Update security controls

    For data breach incidents: 1. Engage legal team immediately 2. Prepare customer communication 3. Follow regulatory notification requirements 4. Document all actions taken

    Scaling and Capacity Planning¶

    Scaling Strategies¶

    Automatic scaling configurations:

    Component Scaling Metric Min Max Scale Up Scale Down
    Web Tier CPU & Request Count 3 30 >70% CPU for 3m <30% CPU for 10m
    API Tier CPU & Queue Depth 5 50 >60% CPU for 2m <40% CPU for 10m
    Worker Tier Queue Depth 2 20 >500 messages <100 messages for 5m
    Database Connection Count & CPU - - Manual scaling -

    Scaling policies: 1. Scale out before scaling up 2. Maintain 30% headroom for unexpected traffic 3. Database scaling requires planned maintenance window

    Load Testing¶

    Regular performance testing:

    1. Monthly load tests in staging environment
    2. Quarterly capacity tests
    3. Performance budget enforcement in CI

    Load test scenarios: - Normal peak traffic (2x average) - Marketing campaign launch (5x average) - Black Friday scenario (10x average) - Recovery from outage (connection storm)

    Resource Planning¶

    Capacity review process:

    1. Weekly usage trend analysis
    2. Monthly capacity planning meeting
    3. Quarterly infrastructure budget review

    Growth-driven scaling triggers: - Approaching 70% of provisioned capacity - Customer growth exceeding 10% month-over-month - New feature launches with significant resource needs

    Routine Maintenance¶

    Scheduled Tasks¶

    Regular maintenance schedule:

    Task Frequency Timing Impact
    Database vacuum Weekly Sunday 2 AM EST None
    Log rotation Daily 1 AM EST None
    Certificate renewal Monthly First Sunday 3 AM EST Brief SSL handshake errors (<1s)
    OS patching Monthly Second Sunday 2 AM EST Rolling restarts
    Infrastructure updates Quarterly Announced 2 weeks in advance Potential brief downtime

    Patch Management¶

    Security patch process:

    1. Critical patches applied within 24 hours after testing
    2. High-severity patches applied within 1 week
    3. Regular patches bundled into monthly maintenance
    4. All patches tested in staging before production

    OS and container updates: - Base images updated monthly - Automatic security patches for critical CVEs - Full rebuild of all images monthly

    Dependency Updates¶

    Third-party dependency update strategy:

    1. Security updates applied immediately
    2. Minor version updates applied monthly
    3. Major version updates evaluated quarterly

    Frontend dependencies: - Updated with each release cycle - Breaking changes evaluated in staging

    Backend dependencies: - Updated during monthly maintenance - Database drivers tested extensively before updating

    Disaster Recovery¶

    Backup Strategy¶

    Data protection approach:

    1. Database: Point-in-time recovery for last 7 days
    2. Object storage: Cross-region replication
    3. Configuration: Version-controlled and backed up
    4. Machine images: Weekly snapshots retained for 30 days

    Recovery point objective (RPO): - Primary data: 15 minutes - Analytics data: 1 hour - Logs and metrics: 24 hours

    Recovery Procedures¶

    Disaster recovery runbooks:

    1. Database Failure: Failover to read replica 

      ops db failover --env=production

    2. Availability Zone Failure: Redirect traffic to functioning AZs 

      ops traffic shift --from=us-east-1a --to=us-east-1b,us-east-1c

    3. Region Failure: Activate cross-region recovery 

      ops dr activate --region=us-west-2

    4. Ransomware/Data Corruption: Restore from clean backups 

      ops restore --service=all --timestamp="2025-03-04T08:00:00Z"

    Recovery time objectives (RTO): - Critical services: 1 hour - Non-critical services: 4 hours - Full system: 8 hours

    Business Continuity¶

    Business continuity planning:

    1. Annual tabletop exercises for major scenarios
    2. Quarterly review of recovery procedures
    3. Emergency contact list maintained in PagerDuty and printed copies
    4. Cross-training of personnel on critical recovery procedures

    Alternative operations modes: - Read-only mode for database issues - Essential services only mode for compute constraints - Manual processing pipeline for queue system issues

    Compliance and Audit¶

    Compliance Requirements¶

    Maily complies with the following standards:

    1. SOC 2 Type II
    2. GDPR
    3. CCPA
    4. ISO 27001
    5. HIPAA (for healthcare customers)

    Compliance responsibilities: - Engineering: Technical controls implementation - Security: Control effectiveness monitoring - Legal: Regulatory requirement tracking - Operations: Evidence collection and documentation

    Audit Logging¶

    Comprehensive audit logging:

    System Events Logged Retention
    Authentication Login, logout, failures 1 year
    Authorization Access attempts, changes 1 year
    Data access PII access, exports, changes 7 years
    Configuration System setting changes 7 years
    Administration User/role management 7 years

    Audit log protection: - Write-only access - Tamper-evident storage - Encryption at rest - Access limited to security team

    Retention Policies¶

    Data lifecycle management:

    1. Customer data: Retained per terms of service
    2. Employee data: Retained per HR policies
    3. Operational data: Retained per compliance requirements
    4. Backup data: Retained per backup strategy

    Deletion verification: - Automated verification of deletion - Certificate of destruction provided on request - Annual data purge verification audit
