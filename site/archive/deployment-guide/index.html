    This document provides comprehensive instructions for deploying the Maily platform to various environments.

    Architecture Overview¶

    Maily is deployed as a set of microservices on Kubernetes, with the following components:

    • Frontend: Next.js application served via Nginx
    • API: Node.js/Express backend services
    • Worker: Background processing services
    • Database: PostgreSQL for persistent storage
    • Redis: For caching and session management
    • RabbitMQ: For message queuing

    The infrastructure is provisioned using Terraform on AWS, with the following key components:

    • EKS for Kubernetes orchestration
    • RDS for managed PostgreSQL
    • ElastiCache for managed Redis
    • Amazon MQ for managed RabbitMQ
    • S3 for object storage
    • CloudFront for CDN
    • Route53 for DNS management
    • WAF for web application firewall

    Prerequisites¶

    Before deploying Maily, ensure you have the following:

    • AWS account with appropriate permissions
    • AWS CLI configured with access credentials
    • Terraform (v1.0.0+)
    • kubectl (compatible with your EKS version)
    • Helm (v3.0.0+)
    • Docker (for local development and testing)

    Infrastructure Setup¶

    Terraform Initialization¶

    1. Navigate to the Terraform directory for your environment:
    cd infrastructure/terraform/environments/[environment]
    1. Initialize Terraform:
    terraform init
    1. Create a terraform.tfvars file with your specific configuration:
    aws_region         = "us-west-2"
vpc_cidr           = "10.0.0.0/16"
availability_zones = ["us-west-2a", "us-west-2b", "us-west-2c"]
    1. Apply the Terraform configuration:
    terraform apply
    1. After successful application, note the outputs for future reference:
    terraform output

    Kubernetes Deployment¶

    Connecting to the Cluster¶

    1. Update your kubeconfig to connect to the EKS cluster:
    aws eks update-kubeconfig --name maily-[environment] --region us-west-2
    1. Verify connection:
    kubectl get nodes

    Deploying with Helm¶

    1. Deploy Maily using Helm:
    helm upgrade --install maily ./infrastructure/helm/maily \
  --namespace maily \
  --create-namespace \
  --set environment=[environment] \
  --values ./infrastructure/helm/maily/values-[environment].yaml
    1. Verify the deployment:
    kubectl get pods -n maily

    Secrets Management¶

    Maily uses HashiCorp Vault for secrets management, integrated with Kubernetes via the External Secrets Operator.

    Setting Up Vault¶

    1. Deploy Vault using Helm:
    helm repo add hashicorp https://helm.releases.hashicorp.com
helm install vault hashicorp/vault --namespace vault --create-namespace

    1. Initialize and unseal Vault (follow Vault documentation for production setup).

    2. Configure Kubernetes authentication:

    vault auth enable kubernetes
vault write auth/kubernetes/config \
  kubernetes_host="https://$KUBERNETES_HOST:443" \
  kubernetes_ca_cert=@/var/run/secrets/kubernetes.io/serviceaccount/ca.crt \
  token_reviewer_jwt=@/var/run/secrets/kubernetes.io/serviceaccount/token
    1. Create a policy and role for Maily:
    vault policy write maily-policy - <<EOF
path "secret/data/maily/*" {
  capabilities = ["read"]
}
EOF

vault write auth/kubernetes/role/maily-role \
  bound_service_account_names=default \
  bound_service_account_namespaces=maily \
  policies=maily-policy \
  ttl=1h
    1. Store secrets in Vault:
    vault kv put secret/maily/database url="postgresql://user:password@hostname:5432/dbname"
vault kv put secret/maily/redis url="redis://hostname:6379"
# Add other secrets as needed
    1. Deploy the External Secrets Operator and configure it to fetch secrets from Vault.

    Monitoring and Observability¶

    Maily uses the Prometheus stack for monitoring and Loki for log aggregation.

    Deploying Prometheus and Grafana¶

    1. Deploy the Prometheus Operator:
    kubectl apply -f infrastructure/kubernetes/monitoring/prometheus-operator.yaml
    1. Deploy Loki for log aggregation:
    kubectl apply -f infrastructure/kubernetes/monitoring/loki.yaml
    1. Access Grafana:
    kubectl port-forward svc/prometheus-operator-grafana 3000:80 -n monitoring
    1. Log in with the default credentials (admin/admin-password-to-be-replaced) and change the password.

    Backup and Disaster Recovery¶

    Maily uses Velero for Kubernetes backup and AWS native services for database backups.

    Setting Up Velero¶

    1. Deploy Velero:
    kubectl apply -f infrastructure/kubernetes/backup/velero.yaml
    1. Verify the installation:
    kubectl get pods -n velero
    1. Create an on-demand backup:
    velero backup create maily-backup --include-namespaces maily

    Database Backups¶

    RDS automated backups are configured via Terraform with a retention period of 7 days. For point-in-time recovery, ensure that automated backups are enabled.

    CI/CD Pipeline¶

    Maily uses GitHub Actions for CI/CD. The workflow is defined in .github/workflows/enhanced-ci-cd.yaml.

    Setting Up GitHub Secrets¶

    The following secrets need to be configured in your GitHub repository:

    • AWS_ACCESS_KEY_ID: AWS access key with appropriate permissions
    • AWS_SECRET_ACCESS_KEY: Corresponding AWS secret key
    • CODECOV_TOKEN: Token for uploading test coverage to Codecov
    • SNYK_TOKEN: Token for Snyk security scanning

    Security Considerations¶

    Network Policies¶

    Apply network policies to restrict pod-to-pod communication:

    kubectl apply -f infrastructure/kubernetes/security/enhanced-network-policies.yaml

    Pod Security Policies¶

    Apply pod security policies to enforce security standards:

    kubectl apply -f infrastructure/kubernetes/security/pod-security-policies.yaml

    Web Application Firewall¶

    The AWS WAF is configured via Terraform to protect against common web exploits.

    Troubleshooting¶

    Common Issues¶

    1. Pod Startup Failures:
    2. Check pod logs: kubectl logs <pod-name> -n maily

    3. Check pod events: kubectl describe pod <pod-name> -n maily

    4. Database Connection Issues:

    5. Verify secrets are correctly mounted
    6. Check network policies allow the connection

    7. Verify RDS security group allows traffic from the EKS cluster

    8. CI/CD Pipeline Failures:

    9. Check GitHub Actions logs for detailed error messages
    10. Verify all required secrets are configured
    11. Ensure AWS credentials have sufficient permissions
