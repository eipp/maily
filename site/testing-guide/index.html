    Overview¶

    This Testing Guide serves as the comprehensive source of truth for all testing activities across the Maily platform. It consolidates previously scattered documentation into a single reference that covers our testing philosophy, methodologies, tools, and best practices.

    Whether you're a new developer looking to understand our testing approach or an experienced team member seeking specific testing patterns, this guide provides the foundation for ensuring quality across our codebase.

    Testing Philosophy¶

    At Maily, we follow these core testing principles:

    1. Shift Left: Catch issues as early as possible in the development lifecycle
    2. Test Pyramid: Prioritize fast, reliable unit tests with fewer integration and E2E tests
    3. Ownership: Developers own quality and testing of their code
    4. Automation First: Automated tests are preferred over manual verification
    5. Value Over Coverage: Focus on testing critical paths and business logic
    6. Maintainability: Tests should be readable, reliable, and easy to maintain

    Test Types¶

    Unit Testing¶

    Unit tests verify individual components, functions, or classes in isolation from their dependencies.

    Key Characteristics: - Fast execution (milliseconds) - Tests single unit of functionality - Dependencies are mocked or stubbed - Should make up ~70% of test suite

    Example Unit Test:

    // Component unit test
describe('EmailTemplateSelect', () => {
  it('should display available templates', () => {
    const templates = [
      { id: '1', name: 'Welcome Email' },
      { id: '2', name: 'Newsletter' }
    ];

    const { getAllByRole } = render(<EmailTemplateSelect templates={templates} />);

    const options = getAllByRole('option');
    expect(options).toHaveLength(2);
    expect(options[0]).toHaveTextContent('Welcome Email');
    expect(options[1]).toHaveTextContent('Newsletter');
  });
});

    Integration Testing¶

    Integration tests verify that multiple components work together correctly.

    Key Characteristics: - Medium execution speed - Tests interactions between components - Some dependencies may be real, others mocked - Should make up ~20% of test suite

    Example Integration Test:

    // Service integration test
describe('Campaign Service', () => {
  let campaignService: CampaignService;
  let mockDbClient: DbClient;
  let mockAnalyticsService: AnalyticsService;

  beforeEach(() => {
    mockDbClient = mock(DbClient);
    mockAnalyticsService = mock(AnalyticsService);
    campaignService = new CampaignService(mockDbClient, mockAnalyticsService);
  });

  it('should create a campaign and track event', async () => {
    // Arrange
    const campaignData = { name: 'Test Campaign', subject: 'Test Subject' };
    when(mockDbClient.insertCampaign).calledWith(expect.anything()).mockResolvedValue({ id: '123', ...campaignData });

    // Act
    const result = await campaignService.createCampaign(campaignData);

    // Assert
    expect(result.id).toBe('123');
    verify(mockAnalyticsService.trackEvent).calledWith('campaign_created', expect.anything());
  });
});

    End-to-End Testing¶

    E2E tests verify the entire application works together from the user's perspective.

    Key Characteristics: - Slower execution (seconds to minutes) - Tests full user workflows across system boundaries - Minimal mocking, uses real services where possible - Should make up ~10% of test suite

    Example E2E Test:

    // E2E test using Playwright
test('user can create and send a campaign', async ({ page }) => {
  // Login
  await page.goto('/login');
  await page.fill('[data-testid="email-input"]', 'test@example.com');
  await page.fill('[data-testid="password-input"]', 'password123');
  await page.click('[data-testid="login-button"]');

  // Create campaign
  await page.click('[data-testid="new-campaign-button"]');
  await page.fill('[data-testid="campaign-name"]', 'My Test Campaign');
  await page.fill('[data-testid="campaign-subject"]', 'Important Announcement');
  await page.click('[data-testid="select-template-button"]');
  await page.click('[data-testid="template-item-newsletter"]');

  // Add content and send
  await page.fill('[data-testid="email-content"]', 'Hello subscribers!');
  await page.click('[data-testid="send-button"]');

  // Verify success
  await expect(page.locator('[data-testid="success-message"]')).toBeVisible();
  await expect(page.locator('[data-testid="campaign-status"]')).toHaveText('Sent');
});

    Performance Testing¶

    Performance tests verify that the application meets performance requirements under expected load.

    Key Tools: - JMeter for API load testing - Lighthouse for frontend performance - Custom instrumentation for service-level metrics

    Performance Test Categories: - Load Testing: System behavior under expected load - Stress Testing: System behavior at or beyond maximum capacity - Endurance Testing: System behavior under sustained load - Spike Testing: System behavior under sudden increases in load

    Security Testing¶

    Security testing identifies vulnerabilities and ensures data protection.

    Automated Security Testing: - Static Application Security Testing (SAST) - Dynamic Application Security Testing (DAST) - Dependency scanning - Secret scanning

    Key Security Test Areas: - Authentication and authorization - Input validation - Data encryption - API security - Dependency vulnerabilities

    Accessibility Testing¶

    Accessibility testing ensures the application is usable by people with disabilities.

    Automated Tools: - axe-core for automated checks - Lighthouse for accessibility audits

    Manual Testing: - Keyboard navigation - Screen reader compatibility - Color contrast verification

    Testing Standards¶

    All tests must follow these standards:

    1. Naming Convention: Tests should clearly describe what is being tested 

      describe('EmailValidator', () => {
  it('should reject emails without @ symbol', () => {...})
});

    2. AAA Pattern: Arrange, Act, Assert 

      // Arrange
const validator = new EmailValidator();
const invalidEmail = 'test.example.com';

// Act
const result = validator.isValid(invalidEmail);

// Assert
expect(result).toBe(false);

    3. Isolation: Tests should not depend on each other

    4. Deterministic: Tests should yield the same results on every run

    5. Focused: Each test should verify a single behavior

    Testing Infrastructure¶

    Test Runners¶

    • Jest: Primary test runner for both frontend and backend unit/integration tests
    • Playwright: End-to-end testing framework
    • Supertest: API testing

    CI/CD Integration¶

    All tests run on CI for: - Pull requests - Main branch merges - Nightly builds (full suite including long-running tests)

    Pipeline Stages: 1. Linting and type checking 2. Unit tests 3. Integration tests 4. E2E tests (subset on PRs, full suite on main) 5. Performance tests (on main only)

    Frontend Testing¶

    Frontend tests focus on components, hooks, state management, and user interactions.

    Component Testing Approach:

    1. Props & Rendering: Verify component renders correctly with different props
    2. User Interactions: Test click, input, and other events
    3. State Changes: Verify component state changes as expected
    4. Edge Cases: Test loading, error, and empty states

    Example Component Test:

    import { render, screen, fireEvent } from '@testing-library/react';
import { SubscriberForm } from './SubscriberForm';

describe('SubscriberForm', () => {
  it('should display validation error for invalid email', () => {
    // Arrange
    render(<SubscriberForm onSubmit={jest.fn()} />);

    // Act
    fireEvent.change(screen.getByLabelText(/email/i), {
      target: { value: 'invalid-email' }
    });
    fireEvent.click(screen.getByRole('button', { name: /save/i }));

    // Assert
    expect(screen.getByText(/please enter a valid email/i)).toBeInTheDocument();
  });
});

    Custom Hooks Testing:

    import { renderHook, act } from '@testing-library/react-hooks';
import { useSubscriberForm } from './useSubscriberForm';

describe('useSubscriberForm', () => {
  it('should update form values', () => {
    // Arrange
    const { result } = renderHook(() => useSubscriberForm());

    // Act
    act(() => {
      result.current.updateField('email', 'test@example.com');
    });

    // Assert
    expect(result.current.values.email).toBe('test@example.com');
  });
});

    Backend Testing¶

    Backend tests focus on API endpoints, services, database interactions, and business logic.

    Service Layer Testing:

    describe('EmailService', () => {
  let emailService: EmailService;
  let mockRepository: EmailRepository;

  beforeEach(() => {
    mockRepository = mock(EmailRepository);
    emailService = new EmailService(mockRepository);
  });

  it('should send email and store delivery record', async () => {
    // Arrange
    const emailData = {
      to: 'recipient@example.com',
      subject: 'Test',
      body: 'Hello world'
    };
    when(mockRepository.saveEmailRecord).calledWith(expect.anything()).mockResolvedValue({ id: '123' });

    // Act
    const result = await emailService.sendEmail(emailData);

    // Assert
    expect(result.success).toBe(true);
    verify(mockRepository.saveEmailRecord).calledWith(expect.objectContaining({
      to: 'recipient@example.com',
      status: 'sent'
    }));
  });
});

    API Testing:

    import request from 'supertest';
import { app } from '../app';
import { authMiddleware } from '../middleware/auth';

// Mock auth middleware for testing
jest.mock('../middleware/auth', () => ({
  authMiddleware: jest.fn((req, res, next) => {
    req.user = { id: 'test-user-id', role: 'admin' };
    next();
  })
}));

describe('Campaign API', () => {
  it('should create a campaign', async () => {
    const response = await request(app)
      .post('/api/campaigns')
      .send({
        name: 'Test Campaign',
        subject: 'Hello Subscribers',
        content: '<p>Welcome to our newsletter</p>'
      });

    expect(response.status).toBe(201);
    expect(response.body).toHaveProperty('id');
    expect(response.body.name).toBe('Test Campaign');
  });
});

    Database Testing:

    We use an in-memory database for testing database interactions:

    import { createConnection } from 'typeorm';
import { SubscriberRepository } from './SubscriberRepository';
import { Subscriber } from './Subscriber.entity';

describe('SubscriberRepository', () => {
  let connection;
  let repository: SubscriberRepository;

  beforeAll(async () => {
    connection = await createConnection({
      type: 'sqlite',
      database: ':memory:',
      entities: [Subscriber],
      synchronize: true
    });
    repository = connection.getCustomRepository(SubscriberRepository);
  });

  afterAll(async () => {
    await connection.close();
  });

  it('should find subscribers by segment', async () => {
    // Arrange
    await repository.save([
      { email: 'user1@example.com', segments: ['newsletter'] },
      { email: 'user2@example.com', segments: ['promotion'] }
    ]);

    // Act
    const results = await repository.findBySegment('newsletter');

    // Assert
    expect(results).toHaveLength(1);
    expect(results[0].email).toBe('user1@example.com');
  });
});

    AI Component Testing¶

    AI components require specialized testing approaches due to their probabilistic nature.

    Strategies for AI Testing:

    1. Golden Dataset Testing: Compare outputs against known good results
    2. Invariant Testing: Check properties that should always be true
    3. Metamorphic Testing: Modify inputs in ways that should have predictable effects on outputs
    4. Adversarial Testing: Test with inputs designed to cause failures

    Example AI Test:

    describe('EmailSubjectSuggestionAI', () => {
  let suggestionService: EmailSubjectSuggestionAI;

  beforeEach(() => {
    suggestionService = new EmailSubjectSuggestionAI();
  });

  it('should generate appropriate length subjects', async () => {
    // Arrange
    const emailContent = 'Welcome to our monthly newsletter with updates about our product...';

    // Act
    const suggestions = await suggestionService.generateSuggestions(emailContent);

    // Assert
    for (const suggestion of suggestions) {
      expect(suggestion.length).toBeGreaterThan(10);
      expect(suggestion.length).toBeLessThan(100);
    }
  });

  it('should maintain key entities from content', async () => {
    // Arrange
    const emailContent = 'Introducing our new Premium Plan with advanced analytics features';

    // Act
    const suggestions = await suggestionService.generateSuggestions(emailContent);

    // Assert
    expect(suggestions.some(s =>
      s.includes('Premium') ||
      s.includes('Plan') ||
      s.includes('analytics')
    )).toBe(true);
  });
});

    Mocking Strategies¶

    External Dependencies¶

    Use the following approaches for mocking external dependencies:

    1. HTTP Clients: Use nock or msw to intercept HTTP requests
    2. Databases: Use in-memory databases or repository mocks
    3. Message Queues: Use in-memory implementations
    4. Time: Use Jest's timer mocks

    Example MSW Setup:

    import { setupServer } from 'msw/node';
import { rest } from 'msw';

const server = setupServer(
  rest.get('https://api.example.com/data', (req, res, ctx) => {
    return res(
      ctx.status(200),
      ctx.json({ items: [{ id: 1, name: 'Test' }] })
    );
  })
);

beforeAll(() => server.listen());
afterEach(() => server.resetHandlers());
afterAll(() => server.close());

    Continuous Integration¶

    Our CI pipeline is configured to:

    1. Run fast tests on every PR
    2. Run the full test suite on main branch merges
    3. Report test results and coverage

    Test Run Configuration:

    # Jest configuration for CI
{
  "preset": "ts-jest",
  "testEnvironment": "node",
  "collectCoverage": true,
  "coverageReporters": ["text", "lcov"],
  "coverageThreshold": {
    "global": {
      "branches": 80,
      "functions": 80,
      "lines": 80,
      "statements": 80
    }
  }
}

    Test Reporting¶

    Test results are reported in multiple formats:

    1. Console Output: For local development
    2. JUnit XML: For CI integration
    3. HTML Reports: For detailed analysis
    4. Coverage Reports: For code coverage visualization

    Troubleshooting¶

    Common Test Failures¶

    Issue Possible Causes Solutions
    Flaky tests Time dependencies, race conditions Use deterministic time mocks, add proper waits
    Failing CI but passing locally Environment differences Containerize tests, use CI debug mode
    Slow tests Too many E2E tests, inefficient setup Move tests down the pyramid, optimize setup/teardown
    Memory leaks Unclosed connections, large test data Add proper cleanup, use memory profiling

    Debugging Tests¶

    Jest Debug Tips:

    # Run a specific test with the debugger
node --inspect-brk node_modules/.bin/jest --runInBand MyComponent.test.ts

# Filter tests by name
jest -t "should handle error state"

# Update snapshots
jest --updateSnapshot

    Playwright Debug Tips:

    # Run with UI mode
npx playwright test --ui

# Run with debug mode
npx playwright test --debug

# Generate traces for failed tests
npx playwright test --trace on
