    Overview¶

    The Maily platform is built on a modern, cloud-native architecture designed for scalability, reliability, and developer velocity. This handbook provides a comprehensive overview of the system's architecture, covering all aspects from frontend to infrastructure.

    The architecture follows these key principles:

    • Modularity: Components are designed with clear boundaries and interfaces
    • Scalability: All components can scale independently based on demand
    • Resilience: The system continues functioning even when parts fail
    • Observability: Comprehensive monitoring and logging throughout
    • Security: Zero-trust approach with defense in depth
    • Developer Experience: Optimized for developer productivity

    System Architecture¶

    High-Level Architecture¶

    Maily follows a microservices architecture organized around business capabilities. The system is divided into these major components:

    1. Frontend Layer: Next.js-based web application and API endpoints
    2. API Gateway: GraphQL-based gateway for aggregating backend services
    3. Microservices: Independent services for specific business domains
    4. Data Layer: Persistent storage using PostgreSQL and Redis
    5. AI Services: Enhanced AI capabilities for content generation and optimization
    6. Infrastructure Layer: Kubernetes-based platform with observability

    Microservices¶

    Maily's backend is composed of these core microservices:

    1. Campaign Service: Manages email campaign creation, scheduling, and execution
    2. Content Service: Handles email content, templates, and assets
    3. Audience Service: Manages recipient lists, segmentation, and subscriptions
    4. Analytics Service: Collects and processes engagement metrics
    5. Notification Service: Manages system notifications and alerts
    6. User Service: Handles user authentication, authorization, and preferences
    7. Billing Service: Manages subscription plans and billing

    Each service: - Has its own database or schema - Exposes a GraphQL API - Scales independently - Is deployed as a containerized application in Kubernetes

    Data Flow¶

    The system follows these data flow patterns:

    1. Command Flow: User actions → API Gateway → Appropriate Microservice → Database → Events
    2. Query Flow: User requests → API Gateway → Relevant Microservice(s) → Database → Response
    3. Event Flow: Service events → Kafka → Consuming Services → Database updates

    Events are used extensively for: - Decoupling services - Implementing complex workflows - Enabling real-time updates - Facilitating eventual consistency

    Frontend Architecture¶

    The frontend architecture is built around Next.js App Router with React Server Components, providing a modern and performant user interface.

    React Server Components¶

    Maily leverages React Server Components (RSC) for several advantages:

    • Reduced Client JavaScript: Server components don't send component code to the client
    • Direct Data Access: Server components can access databases directly
    • Improved Performance: By moving rendering work to the server
    • Progressive Enhancement: Core functionality works without JavaScript

    The frontend follows these patterns: - Server components for data fetching and initial rendering - Client components for interactive elements - Streaming for improved perceived performance - Server actions for form handling

    UI Framework¶

    The UI layer is standardized on:

    • Tailwind CSS: For utility-first styling
    • shadcn/ui: For consistent, accessible component primitives
    • Custom design tokens: For brand consistency
    • Framer Motion: For animations and transitions

    Design principles include: - Accessibility (WCAG 2.1 AA compliance) - Responsive design for all screen sizes - Performance-first approach - Consistent UI patterns

    State Management¶

    State management follows modern best practices:

    • Server State: Managed via TanStack Query for caching, revalidation
    • Client State: Managed with Zustand for global UI state
    • Form State: Handled with React Hook Form and Zod
    • URL State: Used for shareable, bookmarkable UI states

    Backend Architecture¶

    API Design¶

    The Maily API layer follows these principles:

    • GraphQL First: Primary API interface is GraphQL
    • Schema-Driven: Types and resolvers generated from schema
    • Strongly Typed: End-to-end type safety with TypeScript
    • Performance: DataLoader pattern for batching and caching

    API gateway handles: - Authentication - Rate limiting - Request routing - Schema stitching across services

    Database Architecture¶

    Maily uses a polyglot persistence approach:

    • PostgreSQL: Primary relational database for structured data
    • Redis: For caching, session management, rate limiting
    • ClickHouse: For analytics data warehousing
    • S3: For email asset storage

    Database design principles include: - Clear domain boundaries between services - Appropriate indexing strategies - Optimization for common query patterns - Data partitioning for large tables

    Authentication & Authorization¶

    Maily implements:

    • NextAuth.js: For authentication provider integration
    • JWT: For secure, stateless authentication
    • RBAC: Role-based access control
    • ABAC: Attribute-based access control for fine-grained permissions
    • OAuth 2.0/OpenID Connect: For third-party authentication

    AI Services Architecture¶

    Enhanced AI Service¶

    The Enhanced AI Service architecture includes:

    • AI Orchestrator: Coordinates requests between models and services
    • RAG Pipeline: Retrieval-augmented generation for accurate content
    • Model Router: Routes requests to appropriate AI models
    • Content Validator: Ensures generated content meets quality standards
    • Feedback Loop: Improves AI based on user interactions

    ML Model Architecture¶

    The ML pipeline architecture includes:

    • Feature Store: Centralized repository for ML features
    • Model Registry: Versioned storage for trained models
    • Training Pipeline: Automated model training and evaluation
    • Inference Service: Low-latency model serving
    • Monitoring: Performance and drift detection

    Models are: - Version controlled with DVC - Continuously evaluated - A/B tested before full deployment - Monitored for drift and performance degradation

    Infrastructure¶

    Deployment Architecture¶

    Maily's infrastructure is built on these principles:

    • Kubernetes: For container orchestration
    • Helm: For deployment templating and packaging
    • ArgoCD: For GitOps-based continuous deployment
    • Multi-region: For high availability and disaster recovery
    • Infrastructure as Code: Using Terraform for all infrastructure

    Monitoring & Observability¶

    The observability stack includes:

    • Prometheus: For metrics collection
    • Grafana: For dashboards and alerting
    • Loki: For log aggregation
    • Tempo: For distributed tracing
    • OpenTelemetry: For instrumentation
    • Healthchecks: For proactive monitoring

    Scaling Strategy¶

    Maily's scaling strategy encompasses:

    • Horizontal Pod Autoscaling: Based on CPU/memory
    • Custom Metrics Scaling: Based on queue length and request rates
    • Vertical Pod Autoscaling: For resource optimization
    • Database Connection Pooling: For database scaling
    • Read Replicas: For read-heavy workloads
    • Kafka Partitioning: For event processing scalability

    Security Architecture¶

    Zero Trust Model¶

    Maily follows a Zero Trust security model:

    • Identity: Strong authentication for all users and services
    • Devices: Security posture checking
    • Network: Mutual TLS between all services
    • Applications: Runtime application self-protection
    • Data: Encryption at rest and in transit

    Data Protection¶

    Data protection measures include:

    • Encryption: All PII encrypted at rest and in transit
    • Data Classification: Structured approach to sensitive data
    • Access Controls: Least privilege principle
    • Audit Logging: All sensitive data access logged
    • Data Retention: Automatic enforcement of retention policies
    • Compliance: GDPR, CCPA, and industry standards

    Performance Considerations¶

    Key performance optimizations include:

    • Edge Caching: Cloudflare for static assets and API responses
    • Database Optimization: Indexing, query optimization, connection pooling
    • Frontend Performance: Code splitting, lazy loading, image optimization
    • Server Rendering: React Server Components for improved TTI
    • Background Processing: Asynchronous handling of non-critical tasks
    • CDN Integration: For global content delivery

    Evolution & Roadmap¶

    The architecture will evolve according to the Modernization Roadmap, with these key initiatives:

    • Full Microservices Migration: Completing the transition to microservices
    • Edge Computing Expansion: Moving more logic to the edge
    • Advanced AI Integration: Deeper integration of AI throughout the platform
    • Enhanced Real-time Capabilities: WebSockets and real-time collaboration
    • Developer Platform: Expanding API and extension capabilities
