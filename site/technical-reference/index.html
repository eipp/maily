    Overview¶

    This technical reference consolidates essential technical specifications and reference materials for the Maily platform. It serves as the definitive resource for developers working with Maily's APIs, services, and infrastructure components.

    API Reference¶

    Core API¶

    The Maily Core API is the primary interface for programmatically interacting with the platform.

    Base URL¶

    Production: https://api.maily.com/v1
Staging: https://api.staging.maily.com/v1
Development: http://localhost:3001/v1

    Authentication¶

    All API requests require authentication using an API key or OAuth token:

    Authorization: Bearer YOUR_API_TOKEN

    Common Headers¶

    Header Description Required
    Authorization Bearer token for authentication Yes
    Content-Type Request content type (usually application/json) Yes for POST/PUT/PATCH
    Accept Response format (usually application/json) No
    X-Request-ID Client-provided request identifier No
    X-Idempotency-Key Unique key for idempotent requests For idempotent operations

    Primary Endpoints¶

    Endpoint Method Description
    /campaigns GET List campaigns
    /campaigns/{id} GET Get campaign details
    /campaigns POST Create a campaign
    /campaigns/{id} PUT Update a campaign
    /campaigns/{id} DELETE Delete a campaign
    /audiences GET List audiences
    /audiences/{id} GET Get audience details
    /audiences POST Create an audience
    /templates GET List templates
    /templates/{id} GET Get template details
    /analytics/campaigns/{id} GET Get campaign analytics

    Pagination¶

    All list endpoints support pagination with these parameters:

    • page: Page number (default: 1)
    • limit: Items per page (default: 20, max: 100)
    • sort: Field to sort by
    • order: Sort order (asc or desc)

    Response structure:

    {
  "data": [...],
  "meta": {
    "total": 100,
    "page": 1,
    "limit": 20,
    "pages": 5
  },
  "links": {
    "self": "/v1/campaigns?page=1&limit=20",
    "first": "/v1/campaigns?page=1&limit=20",
    "prev": null,
    "next": "/v1/campaigns?page=2&limit=20",
    "last": "/v1/campaigns?page=5&limit=20"
  }
}

    Error Responses¶

    All API errors follow this structure:

    {
  "error": {
    "code": "invalid_request",
    "message": "The request was invalid",
    "details": [
      {
        "field": "name",
        "message": "Name is required"
      }
    ],
    "request_id": "req_123456789"
  }
}

    Common error codes:

    Code HTTP Status Description
    authentication_error 401 Invalid authentication credentials
    authorization_error 403 Insufficient permissions
    not_found 404 Resource not found
    validation_error 422 Invalid request parameters
    rate_limit_exceeded 429 Too many requests
    internal_error 500 Server error

    Webhook API¶

    Webhooks allow you to receive real-time notifications when events occur in your Maily account.

    Webhook Endpoints¶

    Event Type Description
    campaign.sent Campaign was sent
    campaign.delivered Email was delivered
    email.opened Email was opened
    email.clicked Link in email was clicked
    email.bounced Email bounced
    subscriber.created New subscriber added
    subscriber.updated Subscriber information updated
    subscriber.unsubscribed Subscriber unsubscribed

    Webhook Payload¶

    {
  "event": "email.opened",
  "timestamp": "2023-09-15T14:22:45Z",
  "data": {
    "campaign_id": "camp_abc123",
    "email_id": "email_def456",
    "subscriber_id": "sub_789xyz",
    "metadata": {
      "user_agent": "Mozilla/5.0...",
      "ip_address": "192.168.1.1",
      "timestamp": "2023-09-15T14:22:43Z"
    }
  }
}

    Webhook Security¶

    Webhooks are signed using an HMAC signature in the X-Maily-Signature header. Verify the signature using your webhook secret:

    const crypto = require('crypto');

function isValidSignature(payload, signature, secret) {
  const hmac = crypto.createHmac('sha256', secret);
  const computedSignature = hmac.update(payload).digest('hex');
  return crypto.timingSafeEqual(
    Buffer.from(signature),
    Buffer.from(computedSignature)
  );
}

    Database Schema¶

    The Maily platform uses a PostgreSQL database with the following primary schema:

    Core Tables¶

    Organizations¶

    CREATE TABLE organizations (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name VARCHAR(255) NOT NULL,
  slug VARCHAR(63) NOT NULL UNIQUE,
  plan_id VARCHAR(63) NOT NULL,
  billing_email VARCHAR(255) NOT NULL,
  created_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT now(),
  updated_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT now()
);

    Users¶

    CREATE TABLE users (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  email VARCHAR(255) NOT NULL UNIQUE,
  name VARCHAR(255) NOT NULL,
  password_hash VARCHAR(255) NOT NULL,
  two_factor_enabled BOOLEAN NOT NULL DEFAULT false,
  two_factor_secret VARCHAR(255),
  created_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT now(),
  updated_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT now()
);

    Organization Users¶

    CREATE TABLE organization_users (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  organization_id UUID NOT NULL REFERENCES organizations(id) ON DELETE CASCADE,
  user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  role VARCHAR(63) NOT NULL,
  created_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT now(),
  updated_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT now(),
  UNIQUE(organization_id, user_id)
);

    Campaigns¶

    CREATE TABLE campaigns (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  organization_id UUID NOT NULL REFERENCES organizations(id) ON DELETE CASCADE,
  name VARCHAR(255) NOT NULL,
  subject VARCHAR(255) NOT NULL,
  sender_name VARCHAR(255) NOT NULL,
  sender_email VARCHAR(255) NOT NULL,
  content_html TEXT NOT NULL,
  content_text TEXT NOT NULL,
  status VARCHAR(63) NOT NULL DEFAULT 'draft',
  scheduled_at TIMESTAMP WITH TIME ZONE,
  sent_at TIMESTAMP WITH TIME ZONE,
  created_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT now(),
  updated_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT now()
);

    Audience¶

    CREATE TABLE audiences (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  organization_id UUID NOT NULL REFERENCES organizations(id) ON DELETE CASCADE,
  name VARCHAR(255) NOT NULL,
  description TEXT,
  created_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT now(),
  updated_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT now()
);

    Subscribers¶

    CREATE TABLE subscribers (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  organization_id UUID NOT NULL REFERENCES organizations(id) ON DELETE CASCADE,
  email VARCHAR(255) NOT NULL,
  first_name VARCHAR(255),
  last_name VARCHAR(255),
  status VARCHAR(63) NOT NULL DEFAULT 'subscribed',
  metadata JSONB NOT NULL DEFAULT '{}',
  created_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT now(),
  updated_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT now(),
  UNIQUE(organization_id, email)
);

    Audience Subscribers¶

    CREATE TABLE audience_subscribers (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  audience_id UUID NOT NULL REFERENCES audiences(id) ON DELETE CASCADE,
  subscriber_id UUID NOT NULL REFERENCES subscribers(id) ON DELETE CASCADE,
  created_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT now(),
  UNIQUE(audience_id, subscriber_id)
);

    Indexing Strategy¶

    Key indexes for optimizing query performance:

    -- Campaigns indexes
CREATE INDEX idx_campaigns_organization_id ON campaigns(organization_id);
CREATE INDEX idx_campaigns_status ON campaigns(status);
CREATE INDEX idx_campaigns_scheduled_at ON campaigns(scheduled_at) WHERE status = 'scheduled';

-- Subscribers indexes
CREATE INDEX idx_subscribers_organization_id ON subscribers(organization_id);
CREATE INDEX idx_subscribers_email ON subscribers(email);
CREATE INDEX idx_subscribers_status ON subscribers(status);
CREATE INDEX idx_subscribers_metadata_gin ON subscribers USING gin(metadata jsonb_path_ops);

-- Audience indexes
CREATE INDEX idx_audiences_organization_id ON audiences(organization_id);

-- Organization users indexes
CREATE INDEX idx_organization_users_organization_id ON organization_users(organization_id);
CREATE INDEX idx_organization_users_user_id ON organization_users(user_id);

    Event System¶

    Maily uses an event-driven architecture for internal communication between services.

    Event Types¶

    Event Type Producer Consumers Description
    email.queued API Service Email Service Email added to sending queue
    email.sent Email Service Analytics Service Email was sent
    email.delivered Email Service Analytics Service Email was delivered
    email.opened Tracking Service Analytics Service Email was opened
    email.clicked Tracking Service Analytics Service Link in email was clicked
    email.bounced Email Service Analytics, Audience Service Email bounced
    email.complained Email Service Analytics, Audience Service Recipient marked as spam
    subscriber.created API Service Audience Service New subscriber created
    subscriber.updated API Service Audience Service Subscriber updated
    subscriber.unsubscribed API Service, Tracking Service Audience, Analytics Service Subscriber unsubscribed

    Event Schema¶

    interface Event<T> {
  id: string;              // Unique event ID
  type: string;            // Event type
  source: string;          // Service that produced the event
  time: string;            // ISO timestamp
  dataVersion: string;     // Schema version
  organization: string;    // Organization ID
  data: T;                 // Event payload
  metadata: {              // Additional metadata
    correlation_id: string;
    user_id?: string;
  };
}

    Authentication¶

    API Authentication Methods¶

    API Key Authentication¶

    Used for server-to-server integrations:

    Authorization: ApiKey sk_live_abcdef123456

    API keys have the format sk_{environment}_{random_string} where environment is live or test.

    OAuth 2.0¶

    Used for user-facing integrations:

    1. Authorization Request: 

      GET https://auth.maily.com/oauth/authorize
?client_id=YOUR_CLIENT_ID
&redirect_uri=YOUR_REDIRECT_URI
&response_type=code
&scope=campaigns:read audiences:read
&state=YOUR_STATE_VALUE

    2. Token Request: 

      POST https://auth.maily.com/oauth/token
Content-Type: application/x-www-form-urlencoded

grant_type=authorization_code
&client_id=YOUR_CLIENT_ID
&client_secret=YOUR_CLIENT_SECRET
&code=AUTHORIZATION_CODE
&redirect_uri=YOUR_REDIRECT_URI

    3. Token Response: 

      {
  "access_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "token_type": "Bearer",
  "expires_in": 7200,
  "refresh_token": "def456...",
  "scope": "campaigns:read audiences:read"
}

    4. Using the Token: 

      Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...

    JWT Format¶

    Tokens use the following JWT claims:

    {
  "iss": "https://auth.maily.com",
  "sub": "user_123456",
  "aud": "https://api.maily.com",
  "exp": 1631664000,
  "iat": 1631656800,
  "scope": "campaigns:read audiences:read",
  "org_id": "org_987654321"
}

    Authorization¶

    Permission Model¶

    Maily uses a role-based access control (RBAC) system with granular permissions:

    Roles¶

    Role Description
    admin Full access to the organization
    manager Manage campaigns, audiences, and templates
    creator Create and edit campaigns and templates
    analyst View analytics and reports
    viewer View-only access

    Permissions¶

    Permissions follow the format resource:action:

    Permission Description
    campaigns:read View campaigns
    campaigns:create Create campaigns
    campaigns:update Edit campaigns
    campaigns:delete Delete campaigns
    campaigns:send Send campaigns
    audiences:read View audiences
    audiences:create Create audiences
    audiences:update Edit audiences
    audiences:delete Delete audiences
    subscribers:read View subscribers
    subscribers:create Add subscribers
    subscribers:update Edit subscribers
    subscribers:delete Delete subscribers
    templates:read View templates
    templates:create Create templates
    templates:update Edit templates
    templates:delete Delete templates
    analytics:read View analytics
    settings:read View settings
    settings:update Update settings

    Role Permission Mapping¶

    {
  "admin": [
    "*:*"
  ],
  "manager": [
    "campaigns:*",
    "audiences:*",
    "subscribers:*",
    "templates:*",
    "analytics:read",
    "settings:read"
  ],
  "creator": [
    "campaigns:read", "campaigns:create", "campaigns:update",
    "audiences:read",
    "subscribers:read",
    "templates:read", "templates:create", "templates:update",
    "analytics:read"
  ],
  "analyst": [
    "campaigns:read",
    "audiences:read",
    "subscribers:read",
    "templates:read",
    "analytics:read"
  ],
  "viewer": [
    "campaigns:read",
    "audiences:read",
    "subscribers:read",
    "templates:read"
  ]
}

    Rate Limiting¶

    API Rate Limits¶

    Rate limits vary by endpoint and plan:

    Plan General Endpoints Bulk Operations Events/Webhooks
    Free 60 req/min 1000 req/hour 5 req/sec
    Starter 120 req/min 5000 req/hour 10 req/sec
    Professional 300 req/min 10000 req/hour 50 req/sec
    Enterprise 1000 req/min 50000 req/hour 100 req/sec

    Rate Limit Response Headers¶

    X-RateLimit-Limit: 60
X-RateLimit-Remaining: 45
X-RateLimit-Reset: 1631657100

    Rate Limit Exceeded Response¶

    {
  "error": {
    "code": "rate_limit_exceeded",
    "message": "Rate limit exceeded. Please try again later.",
    "details": {
      "limit": 60,
      "reset_at": "2023-09-15T14:45:00Z"
    }
  }
}

    Error Handling¶

    Error Response Structure¶

    {
  "error": {
    "code": "error_code",
    "message": "Human-readable message",
    "details": {}, // Additional error details
    "request_id": "req_123456789" // For debugging and support
  }
}

    Standard Error Codes¶

    Code HTTP Status Description
    invalid_request 400 General invalid request
    invalid_parameters 400 Invalid parameters
    authentication_required 401 Authentication is required
    invalid_credentials 401 Invalid authentication credentials
    access_denied 403 Authorization failed
    not_found 404 Resource not found
    method_not_allowed 405 HTTP method not allowed
    resource_conflict 409 Resource conflict
    resource_gone 410 Resource no longer available
    unsupported_media_type 415 Unsupported media type
    unprocessable_entity 422 Validation failed
    rate_limit_exceeded 429 Rate limit exceeded
    internal_server_error 500 Server error
    service_unavailable 503 Service temporarily unavailable

      {
        "field": "name",

  }
}
