    Overview¶

    This guide serves as the comprehensive reference for integrating with the Maily platform. It consolidates all integration-related documentation and provides developers with the tools, information, and best practices needed to build robust connections with Maily.

    The Maily platform offers multiple integration methods to accommodate different use cases and technical requirements. Whether you're implementing simple API calls, setting up webhooks, or building complex custom integrations, this guide provides the necessary details and examples.

    API Integration¶

    Authentication¶

    All API requests to the Maily platform require authentication. We support two primary authentication methods:

    API Key Authentication¶

    The most common method for server-to-server integrations. API keys can be generated in the Maily dashboard under Settings > API.

    curl -X GET "https://api.maily.com/v1/campaigns" \
  -H "Authorization: Bearer your-api-key"

    API keys should be kept secure and never exposed in client-side code.

    OAuth 2.0 Authentication¶

    For applications acting on behalf of users, OAuth 2.0 authentication is recommended. This requires registering your application in the Maily dashboard.

    // Redirect users to the Maily authorization URL
const authUrl = `https://app.maily.com/oauth/authorize?client_id=${clientId}&redirect_uri=${redirectUri}&response_type=code&scope=read_campaigns,write_subscribers`;

    After the user grants permission, exchange the authorization code for an access token:

    const response = await fetch('https://api.maily.com/oauth/token', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json'
  },
  body: JSON.stringify({
    grant_type: 'authorization_code',
    client_id: clientId,
    client_secret: clientSecret,
    code: authorizationCode,
    redirect_uri: redirectUri
  })
});

const { access_token, refresh_token, expires_in } = await response.json();

    Rate Limits¶

    To ensure platform stability and fair usage, the Maily API implements rate limiting:

    Plan Rate Limit
    Basic 60 requests per minute
    Professional 300 requests per minute
    Enterprise 1,000 requests per minute

    Rate limit headers are included in all API responses:

    X-RateLimit-Limit: 60
X-RateLimit-Remaining: 58
X-RateLimit-Reset: 1615293600

    If you exceed the rate limit, the API will respond with a 429 Too Many Requests status code.

    Error Handling¶

    The Maily API uses standard HTTP status codes and returns detailed error messages in JSON format:

    {
  "error": {
    "code": "invalid_parameter",
    "message": "The subscriber email address is invalid",
    "param": "email",
    "request_id": "req_1234567890"
  }
}

    Common error codes include:

    Status Code Description
    400 Bad Request - Invalid parameters
    401 Unauthorized - Authentication failed
    403 Forbidden - Insufficient permissions
    404 Not Found - Resource doesn't exist
    429 Too Many Requests - Rate limit exceeded
    500 Internal Server Error - Server-side issue

    For robust integration, implement proper error handling and retry logic with exponential backoff for rate limits and temporary errors.

    API Versioning¶

    The Maily API uses date-based versioning to ensure backward compatibility:

    https://api.maily.com/v1/campaigns

    Major versions are incremented when backward-incompatible changes are introduced. We maintain support for previous versions for at least 12 months after a new version is released.

    Version deprecation notices are sent via email and displayed in the dashboard at least 3 months before end-of-life.

    Core Endpoints¶

    Campaigns API¶

    The Campaigns API allows you to create, manage, and monitor email campaigns programmatically.

    List Campaigns¶

    GET /v1/campaigns

    Query parameters:

    Parameter Description
    status Filter by campaign status (draft, scheduled, active, completed)
    limit Number of campaigns to return (default: 25, max: 100)
    offset Pagination offset
    sort Sort field and direction (e.g., created_at:desc)

    Response example:

    {
  "data": [
    {
      "id": "camp_12345",
      "name": "Welcome Series",
      "status": "active",
      "subject": "Welcome to Maily!",
      "created_at": "2023-03-01T12:00:00Z",
      "scheduled_at": "2023-03-02T08:00:00Z",
      "stats": {
        "recipients": 1250,
        "opens": 420,
        "clicks": 65
      }
    },
    // More campaigns...
  ],
  "meta": {
    "total": 18,
    "limit": 25,
    "offset": 0
  }
}

    Create Campaign¶

    POST /v1/campaigns

    Request body:

    {
  "name": "Product Launch",
  "subject": "Introducing Our New Product",
  "template_id": "tmpl_67890",
  "segment_id": "seg_12345",
  "schedule": {
    "send_at": "2023-03-15T09:00:00Z",
    "time_zone": "America/New_York"
  },
  "settings": {
    "from_name": "Marketing Team",
    "from_email": "marketing@example.com",
    "reply_to": "support@example.com"
  },
  "content": {
    "html": "<html><body><h1>Hello {{name}}</h1><p>Check out our new product!</p></body></html>",
    "text": "Hello {{name}}, Check out our new product!"
  }
}

    Get Campaign¶

    GET /v1/campaigns/:id

    Update Campaign¶

    PATCH /v1/campaigns/:id

    Delete Campaign¶

    DELETE /v1/campaigns/:id

    Campaign Actions¶

    POST /v1/campaigns/:id/send    # Send a draft campaign
POST /v1/campaigns/:id/pause   # Pause an active campaign
POST /v1/campaigns/:id/resume  # Resume a paused campaign
POST /v1/campaigns/:id/cancel  # Cancel a scheduled campaign
POST /v1/campaigns/:id/duplicate  # Duplicate a campaign

    Subscribers API¶

    The Subscribers API enables managing contact lists and subscriber data.

    List Subscribers¶

    GET /v1/subscribers

    Query parameters:

    Parameter Description
    segment_id Filter by segment
    status Filter by status (active, unsubscribed, bounced)
    email Search by email address
    limit Number of subscribers to return (default: 25, max: 100)
    offset Pagination offset

    Create/Update Subscriber¶

    POST /v1/subscribers

    Request body:

    {
  "email": "user@example.com",
  "first_name": "John",
  "last_name": "Doe",
  "status": "active",
  "custom_fields": {
    "company": "Acme Inc",
    "industry": "Technology",
    "plan_level": "premium"
  },
  "tags": ["new-user", "product-interest-a"]
}

    Bulk Operations¶

    POST /v1/subscribers/batch

    Request body:

    {
  "subscribers": [
    {
      "email": "user1@example.com",
      "first_name": "Jane"
    },
    {
      "email": "user2@example.com",
      "first_name": "Bob"
    }
  ],
  "update_existing": true
}

    Subscriber Actions¶

    POST /v1/subscribers/:id/unsubscribe  # Unsubscribe a subscriber
POST /v1/subscribers/:id/resubscribe  # Resubscribe a subscriber

    Templates API¶

    The Templates API allows managing email templates programmatically.

    List Templates¶

    GET /v1/templates

    Create Template¶

    POST /v1/templates

    Request body:

    {
  "name": "Monthly Newsletter",
  "category": "newsletters",
  "content": {
    "html": "<html><body><h1>Monthly Update</h1>{{content}}</body></html>",
    "text": "Monthly Update\n\n{{content}}"
  },
  "variables": ["content", "user_name", "products"]
}

    Template Operations¶

    GET /v1/templates/:id          # Get template details
PATCH /v1/templates/:id        # Update template
DELETE /v1/templates/:id       # Delete template
POST /v1/templates/:id/test    # Send test email

    "delivery_rate": 99.0,
    "open_rate": 37.0,
    "click_rate": 13.0,

    {
      "date": "2023-03-01T00:00:00Z",

}

    Subscriber Analytics¶

    GET /v1/analytics/subscribers/:id

    Response includes engagement history, campaign interactions, and activity timeline.

    SDK Integrations¶

    JavaScript SDK¶

    The Maily JavaScript SDK provides a convenient way to integrate with the Maily API from browser or Node.js applications.

    Installation¶

    npm install @maily/sdk
# or
yarn add @maily/sdk

    Usage¶

    import { MailyClient } from '@maily/sdk';

// Initialize the client
const maily = new MailyClient({
  apiKey: 'your-api-key',
  version: 'v1'
});

// Create a subscriber
async function addSubscriber() {
  try {
    const subscriber = await maily.subscribers.create({
      email: 'user@example.com',
      first_name: 'John',
      last_name: 'Doe',
      custom_fields: {
        company: 'Acme Inc'
      }
    });

    console.log('Subscriber created:', subscriber.id);
  } catch (error) {
    console.error('Error creating subscriber:', error.message);
  }
}

// Get campaign analytics
async function getCampaignStats(campaignId) {
  try {
    const stats = await maily.analytics.getCampaign(campaignId, {
      metrics: ['opens', 'clicks', 'bounces'],
      interval: 'daily'
    });

    console.log('Campaign stats:', stats);
  } catch (error) {
    console.error('Error fetching stats:', error.message);
  }
}

    PHP SDK¶

    Installation¶

    composer require maily/maily-php

    Usage¶

    <?php
require 'vendor/autoload.php';

$maily = new Maily\MailyClient('your-api-key');

// Get a list of templates
$templates = $maily->templates->list([
    'category' => 'newsletters',

            ],

            ]
        ]
    ]
]);

echo "Created segment with ID: " . $segment->id;

    Webhook Events¶

    Webhooks allow your application to receive real-time notifications about events in the Maily platform.

    Event Structure¶

    All webhook events share a common structure:

    {
  "id": "evt_1234567890",
  "created_at": "2023-03-01T12:00:00Z",
  "type": "campaign.sent",
  "data": {
    // Event-specific data
  }
}

    Event Types¶

    Campaign Events¶

    • campaign.created - Campaign created
    • campaign.updated - Campaign updated
    • campaign.sent - Campaign sent
    • campaign.completed - Campaign delivery completed
    • campaign.canceled - Campaign canceled

    Subscriber Events¶

    • subscriber.created - New subscriber added
    • subscriber.updated - Subscriber data updated
    • subscriber.unsubscribed - Subscriber unsubscribed
    • subscriber.bounced - Email to subscriber bounced
    • subscriber.complained - Subscriber reported spam

    Email Events¶

    • email.sent - Email sent
    • email.delivered - Email delivered
    • email.opened - Email opened
    • email.clicked - Link in email clicked
    • email.bounced - Email bounced
    • email.dropped - Email dropped (invalid address, etc.)

    Webhook Security¶

    To ensure the security of webhook events, we sign all webhook payloads. The signature is included in the X-Maily-Signature header.

    const crypto = require('crypto');

function verifyWebhookSignature(payload, signature, secret) {
  const expectedSignature = crypto
    .createHmac('sha256', secret)
    .update(payload)
    .digest('hex');

  return crypto.timingSafeEqual(
    Buffer.from(signature),
    Buffer.from(expectedSignature)
  );
}

// In your webhook handler
function handleWebhook(req, res) {
  const payload = req.body;
  const signature = req.headers['x-maily-signature'];

  if (!verifyWebhookSignature(JSON.stringify(payload), signature, webhookSecret)) {
    return res.status(401).send('Invalid signature');
  }

  // Process the webhook event
  const event = payload;
  console.log(`Received event: ${event.type}`);

  // Handle different event types
  switch (event.type) {
    case 'campaign.sent':
      handleCampaignSent(event.data);
      break;
    case 'email.opened':
      handleEmailOpened(event.data);
      break;
    // Handle other event types...
  }

  res.status(200).send('Webhook received');
}

    Retries and Failures¶

    If your webhook endpoint returns a non-200 response or times out, we will retry the delivery using the following schedule:

    • 1st retry: 5 minutes after the initial attempt
    • 2nd retry: 30 minutes after the initial attempt
    • 3rd retry: 2 hours after the initial attempt
    • 4th retry: 6 hours after the initial attempt
    • 5th retry: 24 hours after the initial attempt

    After the 5th retry, the event will be marked as failed and will not be retried again.

    You can view failed webhook deliveries in the Maily dashboard and manually retry them if needed.

    Single Sign-On¶

    SAML Integration¶

    Maily supports SAML 2.0 for enterprise single sign-on integration.

    Configuration Steps¶

    1. Exchange Metadata
    2. Provide your Identity Provider (IdP) metadata to Maily

    3. Download Maily's Service Provider (SP) metadata

    4. Configure IdP Settings

    5. Entity ID: https://auth.maily.com/saml2
    6. ACS URL: https://auth.maily.com/saml2/acs

    7. RelayState: Optional, for deep linking

    8. Attribute Mapping

    9. Email: email or http://schemas.xmlsoap.org/ws/2005/05/identity/claims/emailaddress
    10. First Name: firstName or http://schemas.xmlsoap.org/ws/2005/05/identity/claims/givenname
    11. Last Name: lastName or http://schemas.xmlsoap.org/ws/2005/05/identity/claims/surname
    12. Role: role or http://schemas.microsoft.com/ws/2008/06/identity/claims/role

    Testing and Troubleshooting¶

    • Initiate a test login from the Maily dashboard
    • Check IdP logs for authentication failures
    • Verify certificate expiration dates
    • Ensure clock synchronization between IdP and Maily

    OAuth 2.0¶

    For applications that need to allow users to authorize access to their Maily account.

    Registration¶

    Register your application in the Maily dashboard to receive a client ID and secret.

    Authorization Flow¶

    1. Authorization Request
    GET https://app.maily.com/oauth/authorize

    Parameters: - client_id: Your application's client ID - redirect_uri: URL to redirect back to after authorization - response_type: Set to code - scope: Space-separated list of permissions - state: Random string to prevent CSRF attacks

    1. Token Exchange
    POST https://api.maily.com/oauth/token

    Request body: - grant_type: authorization_code - client_id: Your application's client ID - client_secret: Your application's client secret - code: The authorization code received - redirect_uri: Must match the original redirect URI

    Response:

    {
  "access_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "token_type": "Bearer",
  "expires_in": 7200,
  "refresh_token": "def50200b97...",
  "scope": "read_campaigns write_subscribers"
}
    1. Using the Access Token
    curl -X GET "https://api.maily.com/v1/campaigns" \
  -H "Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."
    1. Refreshing Tokens
    POST https://api.maily.com/oauth/token

    Request body: - grant_type: refresh_token - client_id: Your application's client ID - client_secret: Your application's client secret - refresh_token: The refresh token

    JWT Authentication¶

    For server-to-server integrations, JWT authentication provides a stateless authentication mechanism.

    Creating a JWT¶

    const jwt = require('jsonwebtoken');

function generateJWT(apiKey, apiSecret) {
  const now = Math.floor(Date.now() / 1000);

  const payload = {
    iss: apiKey,
    iat: now,
    exp: now + 300, // 5 minutes expiration
    jti: randomUUID() // Unique token ID
  };

  return jwt.sign(payload, apiSecret, { algorithm: 'HS256' });
}

// Use the JWT for authentication
const token = generateJWT('api_key', 'api_secret');

fetch('https://api.maily.com/v1/campaigns', {

    Nango Integration Platform¶

    Maily uses the Nango open-source API integration platform to enable seamless connection to third-party platforms like LinkedIn, Twitter, Gmail, and Outlook. This significantly reduces the development time and maintenance burden for multi-platform integrations.

    Benefits of Nango Integration¶

    1. Streamlined OAuth Implementation: Handles complex OAuth flows for various platforms, reducing authentication code by 80-90%.
    2. Unified Token Management: Securely stores, encrypts, and automatically refreshes access tokens.
    3. Normalized API Access: Provides consistent interfaces across different platforms.
    4. Built-in Syncing: Handles data synchronization logic across platforms.
    5. Development Speed: Reduces integration development time by 40-50%.

    Architecture¶

    The Nango integration in Maily consists of several components:

    1. Backend Services:
    2. NangoIntegrationService: Core service for interacting with the Nango API
    3. PlatformIntegrationOrchestrator: Orchestrates platform integrations using Nango and OctoTools

    4. IdentityResolutionService: Resolves identities across different platforms

    5. API Routes:

    6. /api/integrations/connect: Start the platform connection flow
    7. /api/integrations/callback: Handle OAuth callbacks
    8. /api/integrations/connections: List connected platforms
    9. /api/integrations/sync: Trigger synchronization
    10. /api/integrations/{platform}: Disconnect a platform
    11. /api/integrations/{platform}/{sync_type}/data: Get synchronized data

    Supported Platforms¶

    1. LinkedIn
    2. Search for people, send messages, access connections

    3. Synchronizes contacts, companies, and posts

    4. Twitter

    5. Post tweets, search for tweets, access followers

    6. Synchronizes followers and tweets

    7. Gmail

    8. Send emails, search for emails, access contacts

    9. Synchronizes contacts and messages

    10. Outlook

    11. Send emails, search for emails, access contacts
    12. Synchronizes contacts and messages

    Identity Resolution¶

    The IdentityResolutionService provides cross-platform identity resolution, allowing Maily to:

    1. Match contacts across different platforms
    2. Create a unified view of contacts
    3. Assign confidence scores to matches
    4. Generate global identifiers for contacts

    Implementation¶

    To implement Nango integration:

    // Connect to a platform
const response = await fetch('/api/integrations/connect', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json'
  },
  body: JSON.stringify({
    platform: 'linkedin',
    redirectUrl: window.location.href
  })
});

const { authUrl } = await response.json();
window.location.href = authUrl;

// After authentication, list connected platforms
const connections = await fetch('/api/integrations/connections').then(res => res.json());

// Trigger a sync
await fetch('/api/integrations/sync', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json'
  },
  body: JSON.stringify({
    platform: 'linkedin',
    syncType: 'contacts'
  })
});

// Get synchronized data
const contacts = await fetch('/api/integrations/linkedin/contacts/data').then(res => res.json());
