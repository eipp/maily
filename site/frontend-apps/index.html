    Overview¶

    Maily's frontend ecosystem consists of several Next.js applications designed to provide specialized functionality for different user roles and use cases. This document provides comprehensive documentation for all frontend applications within the Maily platform.

    Our applications are built on a shared foundation of UI components, utilities, and services, providing a consistent user experience while maintaining separation of concerns between different functional areas.

    Application Architecture¶

    All Maily frontend applications follow a consistent architectural pattern:

    Technology Stack¶

    • Framework: Next.js 14+ with App Router
    • UI Library: React 18+
    • State Management: React Context API, Zustand, and React Query
    • Styling: Tailwind CSS with CSS Modules
    • API Communication: GraphQL with Apollo Client
    • Form Management: React Hook Form with Zod validation
    • Internationalization: next-intl
    • Testing: Vitest, React Testing Library, and Playwright

    Architectural Principles¶

    1. Component-Driven Development: Building interfaces from small, reusable components
    2. Server-First Approach: Leveraging React Server Components for performance
    3. Progressive Enhancement: Ensuring functionality with and without JavaScript
    4. Responsive Design: Supporting all device types and screen sizes
    5. Accessibility First: Ensuring WCAG 2.1 AA compliance from the beginning
    6. Performance Budgets: Setting and enforcing strict performance metrics
    7. Type Safety: Comprehensive TypeScript typing throughout the codebase

    Folder Structure¶

    All applications follow a standardized project structure: 

    ├── app/                  # Next.js App Router structure
│   ├── [locale]/         # Internationalization root
│   │   ├── layout.tsx    # Root layout with providers
│   │   ├── page.tsx      # Home page
│   │   ├── (routes)/     # Grouped routes
├── components/           # Shared components
│   ├── ui/               # Primitive UI components
│   ├── features/         # Feature-specific components
│   ├── layouts/          # Layout components
├── lib/                  # Shared utilities
│   ├── api/              # API client and hooks
│   ├── utils/            # Utility functions
│   ├── hooks/            # Custom hooks
├── public/               # Static assets
├── styles/               # Global styles
├── middleware.ts         # Next.js middleware
├── next.config.js        # Next.js configuration
├── package.json          # Dependencies and scripts

    Development Workflow¶

    Local Development¶

    To set up a local development environment for any Maily frontend application:

    1. Clone the repository: 

      git clone https://github.com/maily/[app-name].git
cd [app-name]

    2. Install dependencies: 

      pnpm install

    3. Set up environment variables: 

      cp .env.example .env.local
# Edit .env.local with appropriate values

    4. Start the development server: 

      pnpm dev

    5. Access the application at http://localhost:3000

    Testing Strategy¶

    All frontend applications implement a comprehensive testing strategy:

    • Unit Tests: Individual component and utility testing
    • Integration Tests: Component interaction testing
    • End-to-End Tests: Critical user flow testing
    • Visual Regression Tests: UI appearance testing
    • Accessibility Tests: A11y compliance testing
    • Performance Tests: Load time and interaction testing

    Test commands: 

    # Run unit and integration tests
pnpm test

# Run end-to-end tests
pnpm test:e2e

# Run visual regression tests
pnpm test:visual

# Run accessibility tests
pnpm test:a11y

# Run performance tests
pnpm test:performance

    Deployment Strategy¶

    Environments¶

    All frontend applications are deployed to multiple environments:

    • Development: CI branch deployments for feature testing
    • Staging: Pre-production environment for QA and testing
    • Production: User-facing production environment

    Deployment Mechanism¶

    • Infrastructure: Kubernetes with ArgoCD
    • Hosting: AWS with CloudFront CDN
    • Container: Docker containers with CI builds
    • Configuration: Environment-specific configuration injection
    • Scaling: Horizontal pod scaling based on load

    Release Strategy¶

    • Release Cadence: Varies by application (weekly to monthly)
    • Hotfix Process: Critical fixes bypass standard cycle
    • Feature Flags: New features controlled by flags
    • Rollout Strategy: Incremental rollout with monitoring
    • Rollback Process: Automated rollback on error thresholds

    Performance Optimization¶

    All frontend applications implement performance best practices:

    Core Web Vitals¶

    • LCP (Largest Contentful Paint): Target < 2.5s (currently 1.2s)
    • FID (First Input Delay): Target < 100ms (currently 45ms)
    • CLS (Cumulative Layout Shift): Target < 0.1 (currently 0.05)

    Optimization Techniques¶

    • Server Components: Leveraging React Server Components for reduced client JS
    • Edge Computing: Deploying critical handlers to the edge
    • Caching Strategy: Aggressive caching with revalidation
    • Code Splitting: Route-based and component-based splitting
    • Image Optimization: Using Next.js Image component with optimization
    • Font Optimization: Self-hosted fonts with display swap
    • Script Optimization: Third-party script loading optimization
    • Static Generation: Pre-rendering where appropriate

    Performance Monitoring¶

    • Real User Monitoring: Tracking real-world performance metrics
    • Synthetic Monitoring: Regular performance testing
    • Lighthouse CI: Performance regression prevention
    • Bundle Analysis: Regular JS bundle size analysis
    • Performance Budget: Enforced size and timing budgets

    Accessibility Compliance¶

    All frontend applications maintain WCAG 2.1 AA compliance:

    Accessibility Features¶

    • Semantic HTML: Proper document structure
    • Keyboard Navigation: Full keyboard control
    • Screen Reader Support: ARIA attributes and label implementation
    • Focus Management: Visible focus indicators and logical tab order
    • Color Contrast: Minimum 4.5:1 contrast ratio
    • Text Resizing: Support for 200% text size
    • Reduced Motion: Support for prefers-reduced-motion
    • Form Accessibility: Proper labeling and error states
    • Alternative Text: Descriptive text for all images

    Testing and Monitoring¶

    • Automated Testing: axe-core integration for automated testing
    • Manual Testing: Regular audits with screen readers
    • Accessibility Statement: Published accessibility commitment
    • Feedback Mechanism: User feedback channel for accessibility issues
    • Regular Audits: Quarterly professional accessibility audits
