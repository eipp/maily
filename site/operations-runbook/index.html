    Overview¶

    This Operations Runbook provides comprehensive procedures and guidelines for the operation, maintenance, and troubleshooting of the Maily platform. It serves as the primary reference for DevOps, SRE, and operations teams.

    The runbook adheres to these core operational principles:

    • Automation First: Wherever possible, operations are automated to reduce manual effort and human error
    • Infrastructure as Code: All infrastructure changes follow the IaC approach
    • Immutable Infrastructure: Services are redeployed rather than modified in place
    • Proactive Monitoring: Anticipating issues before they affect users
    • Continuous Improvement: Regular review and refinement of operational procedures

    Service Management¶

    Deployment Procedures¶

    Standard Deployment¶

    The deployment process uses GitOps methodology with ArgoCD:

    1. Code changes are merged to the main branch after passing all CI checks
    2. The deployment pipeline builds container images and tags them with Git SHA
    3. ArgoCD detects the change in the Git repository and synchronizes the applications
    # Manual deployment (if needed):
argocd app sync maily-web
argocd app sync maily-api
argocd app sync maily-worker

    Deployment Verification¶

    After deployment, automated smoke tests verify the service health:

    1. HTTP health checks confirm services are responding
    2. Functional tests verify key workflows
    3. Metrics are checked for any anomalies
    # Manual verification:
kubectl get pods -n maily-production
curl -s https://api.maily.example.com/health | jq
# Run smoke tests:
pnpm test:smoke

    Release Cadence¶

    • Regular releases: Every two weeks (Sprint cadence)
    • Hotfixes: As needed for critical issues
    • Feature flags: Used to control rollout of new features

    Service Updates¶

    Update Types¶

    • Minor Updates: Regular feature additions and bug fixes
    • Major Updates: Significant architecture or interface changes
    • Dependency Updates: Updates to libraries and dependencies
    • Security Patches: Critical security fixes

    Update Process¶

    1. Create an update plan with rollback strategy
    2. Schedule update during appropriate maintenance window
    3. Deploy to staging environment first
    4. Verify functionality in staging
    5. Deploy to production using blue/green or canary strategy
    6. Monitor application performance and error rates
    7. Confirm update success or trigger rollback

    Dependency Management¶

    • Dependencies are updated on a regular schedule
    • Security patches are prioritized
    • Dependency changes are tested thoroughly before deployment
    • Major version upgrades undergo additional testing

    Rollback Procedures¶

    Rollback Criteria¶

    Rollback should be initiated if any of these conditions are met: - Error rate increases above 1% - Latency increases by 100% or more - Critical business functionality is impaired - Security vulnerability is introduced

    Rollback Process¶

    1. Identify the need to rollback based on monitoring or manual verification
    2. Execute the rollback command to revert to previous stable version
    # Rollback using ArgoCD:
argocd app history maily-api
argocd app rollback maily-api <VERSION>

# Rollback using Helm:
helm rollback maily-api <REVISION>
    1. Verify system stability after rollback
    2. Notify stakeholders of the rollback
    3. Document the issue and create post-mortem

    Rollback Testing¶

    Rollback procedures are tested regularly during non-production deployments to ensure reliability.

    Version Management¶

    Versioning Strategy¶

    Maily uses semantic versioning (MAJOR.MINOR.PATCH): - MAJOR: Incompatible API changes - MINOR: New functionality in a backward compatible manner - PATCH: Backward compatible bug fixes

    Version Tracking¶

    • Git tags mark released versions
    • Container images are tagged with Git SHA and semantic version
    • Version information is available in the application UI and API
    • Release notes document changes in each version

    Artifact Management¶

    • Container images are stored in ECR
    • Helm charts are stored in Artifact Hub
    • Build artifacts are stored in S3
    • All artifacts have retention policies

    Monitoring & Alerting¶

    Monitoring Infrastructure¶

    Components¶

    The monitoring stack consists of: - Prometheus: Metrics collection and alerting - Grafana: Visualization and dashboards - Loki: Log aggregation - Tempo: Distributed tracing - OpenTelemetry: Instrumentation and data collection - AlertManager: Alert routing and notification

    Health Checks¶

    Each service exposes health endpoints: - /health: Basic application health - /health/ready: Readiness check - /health/live: Liveness check - /metrics: Prometheus metrics

    # Check API health
curl -s https://api.maily.example.com/health | jq

# Check database connectivity
kubectl exec -it maily-api-pod-name -- curl localhost:8000/health/db

    Metrics Collection¶

    Key metrics collected include: - Request rate, error rate, and duration (RED) - Resource utilization (CPU, memory, disk, network) - Business metrics (users, campaigns, emails sent) - SLO/SLI metrics (availability, latency)

    Alert Management¶

    Alert Configuration¶

    Alerts are defined in Prometheus using AlertManager:

    # Example alert rule
groups:
- name: maily-api
  rules:
  - alert: HighErrorRate
    expr: sum(rate(http_requests_total{job="maily-api",status=~"5.."}[5m])) / sum(rate(http_requests_total{job="maily-api"}[5m])) > 0.01
    for: 5m
    labels:
      severity: critical
    annotations:
      summary: "High error rate on API"
      description: "Error rate is {{ $value | humanizePercentage }} for the last 5 minutes"

    Alert Severity Levels¶

    • Critical: Immediate action required, customer impact
    • Warning: Action required soon, potential impact
    • Info: No immediate action, potential issue to watch

    Response Times¶

    • Critical alerts: 15-minute response time, 24/7
    • Warning alerts: 4-hour response time during business hours
    • Info alerts: Next business day

    Notification Channels¶

    • PagerDuty for critical alerts
    • Slack for warning and info alerts
    • Email for daily/weekly digests
    • SMS for critical infrastructure failures

    Troubleshooting¶

    Common Issues¶

    API Service Unavailable¶
    1. Check pod status: kubectl get pods -n maily-production
    2. Check logs: kubectl logs -n maily-production deployment/maily-api
    3. Verify dependencies (database, Redis, etc.)
    4. Check for recent deployments or changes
    High Latency¶
    1. Check database query performance
    2. Verify autoscaling is functioning
    3. Check for resource constraints
    4. Review recent code changes
    5. Check external service dependencies
    Elevated Error Rates¶
    1. Identify error types in logs
    2. Check for recent deployments
    3. Verify configuration changes
    4. Check external dependencies
    5. Review recent data changes

    Debugging Tools¶

    • kubectl: Kubernetes management
    • k9s: Interactive Kubernetes UI
    • Grafana Explore: Ad-hoc query interface
    • pgHero: Database performance monitoring
    • New Relic APM: Application performance monitoring

    Log Analysis¶

    • Centralized logging with Loki
    • Structured JSON log format
    • Correlation IDs for request tracing
    • Log levels: ERROR, WARN, INFO, DEBUG
    • Sensitive data is filtered from logs
    # Search logs for a specific request ID
logcli query '{app="maily-api"}' | grep "request_id=abc123"

# Get error logs from the past hour
logcli query '{app="maily-api", level="error"} | json' --since=1h

    Database Operations¶

    Backup & Restore¶

    Backup Procedures¶

    Daily Automated Backups 

    # AWS RDS Automated Backup Configuration
aws rds modify-db-instance \
  --db-instance-identifier maily-prod \
  --backup-retention-period 35 \
  --preferred-backup-window "00:00-01:00" \
  --copy-tags-to-snapshot \
  --deletion-protection \
  --apply-immediately

    Backup Retention - Daily backups: 35 days retention - Weekly backups: 52 weeks retention - Monthly backups: 12 months retention - Cross-region replicas: Real-time replication

    Restore Procedures¶

    Full Database Restore 

    # Restore from snapshot to new instance
aws rds restore-db-instance-from-db-snapshot \
  --db-instance-identifier maily-prod-restore \
  --db-snapshot-identifier rds:maily-prod-snapshot \
  --db-instance-class db.r5.xlarge

# For point-in-time recovery
aws rds restore-db-instance-to-point-in-time \
  --source-db-instance-identifier maily-prod \
  --target-db-instance-identifier maily-prod-restore \
  --restore-time 2025-02-01T00:00:00Z

    Table-Level Restore 1. Restore full database to temporary instance 2. Export specific table data 3. Import data to production database

    Backup Verification¶

    Automated daily verification: - Restore sample from backup to test instance - Run integrity checks - Verify data consistency - Clean up test instance

    Performance Tuning¶

    Query Optimization¶

    • Slow query log analysis
    • Index optimization
    • Query rewriting
    • Execution plan analysis
    -- Enable slow query logging
ALTER SYSTEM SET log_min_duration_statement = 1000; -- 1 second
ALTER SYSTEM SET log_statement = 'none';
SELECT pg_reload_conf();

-- Find missing indexes
SELECT
  relname,
  seq_scan - idx_scan AS missed_scans,
  round(100 * seq_scan / GREATEST(seq_scan + idx_scan, 1))::numeric AS seq_pct,
  seq_scan, idx_scan
FROM pg_stat_user_tables
WHERE seq_scan > 0
ORDER BY missed_scans DESC;

    Connection Pooling¶

    PgBouncer is configured for connection pooling: - Pool mode: transaction - Default pool size: 20 - Max client connections: 1000

    # pgbouncer.ini
[databases]
* = host=maily-prod.example.com port=5432 dbname=maily

[pgbouncer]
listen_addr = *
listen_port = 6432
auth_type = md5
auth_file = /etc/pgbouncer/userlist.txt
pool_mode = transaction
default_pool_size = 20
max_client_conn = 1000

    Maintenance Tasks¶

    Regular Maintenance¶

    • VACUUM ANALYZE: Daily automated job
    • Database statistics update: Daily
    • Index maintenance: Weekly
    • Storage cleanup: Monthly
    -- Regular maintenance tasks
VACUUM ANALYZE;

-- Update optimizer statistics
ANALYZE VERBOSE;

-- Rebuild indexes
REINDEX TABLE users;

-- Find and remove bloat
SELECT schemaname, tablename,
  pg_size_pretty(pg_table_size(schemaname || '.' || tablename)) as table_size,
  pg_size_pretty(bloat_size) as bloat_size,
  round(100 * bloat_size / pg_table_size(schemaname || '.' || tablename)) as bloat_percentage
FROM (
  -- Bloat estimation query
  -- ...complex query omitted for brevity...
) AS bloat_estimation
WHERE bloat_percentage > 20
ORDER BY bloat_percentage DESC;

    Health Checks¶

    Daily automated health checks: - Connection count monitoring - Lock monitoring - Transaction wraparound prevention - Replication lag monitoring

    -- Check for long-running transactions
SELECT pid, now() - pg_stat_activity.query_start AS duration, query
FROM pg_stat_activity
WHERE state = 'active' AND now() - pg_stat_activity.query_start > interval '5 minutes'
ORDER BY duration DESC;

-- Check for locks
SELECT blocked_locks.pid AS blocked_pid,
       blocked_activity.usename AS blocked_user,
       blocking_locks.pid AS blocking_pid,
       blocking_activity.usename AS blocking_user,
       blocked_activity.query AS blocked_statement,
       blocking_activity.query AS blocking_statement
FROM pg_catalog.pg_locks blocked_locks
JOIN pg_catalog.pg_stat_activity blocked_activity ON blocked_activity.pid = blocked_locks.pid
JOIN pg_catalog.pg_locks blocking_locks
    ON blocking_locks.locktype = blocked_locks.locktype
    AND blocking_locks.DATABASE IS NOT DISTINCT FROM blocked_locks.DATABASE
    AND blocking_locks.relation IS NOT DISTINCT FROM blocked_locks.relation
    AND blocking_locks.page IS NOT DISTINCT FROM blocked_locks.page
    AND blocking_locks.tuple IS NOT DISTINCT FROM blocked_locks.tuple
    AND blocking_locks.virtualxid IS NOT DISTINCT FROM blocked_locks.virtualxid
    AND blocking_locks.transactionid IS NOT DISTINCT FROM blocked_locks.transactionid
    AND blocking_locks.classid IS NOT DISTINCT FROM blocked_locks.classid
    AND blocking_locks.objid IS NOT DISTINCT FROM blocked_locks.objid
    AND blocking_locks.objsubid IS NOT DISTINCT FROM blocked_locks.objsubid
    AND blocking_locks.pid != blocked_locks.pid
JOIN pg_catalog.pg_stat_activity blocking_activity ON blocking_activity.pid = blocking_locks.pid
WHERE NOT blocked_locks.GRANTED;

    Migration Procedures¶

    Schema Migrations¶

    Migrations are handled using Alembic: 1. Development: Create migration in dev environment 2. Testing: Apply and test in staging environment 3. Production: Apply during low-traffic maintenance window

    # Generate new migration
alembic revision --autogenerate -m "Add user preferences table"

# Apply migration
alembic upgrade head

# Downgrade if needed
alembic downgrade -1

    Large Data Migrations¶

    For large tables or data transformations: 1. Create new table structure 2. Implement batch migration process 3. Validate data 4. Switch over to new structure 5. Clean up old data after verification period

    Rollback Planning¶

    Every migration includes a rollback plan: - Reverse migrations defined in Alembic - Data backup before migration - Validation steps - Monitoring during migration

    • P4: Low - Minor impact, no service disruption

    Severity Response Time Resolution Target Update Frequency
    P1 15 minutes 4 hours 30 minutes
    P2 30 minutes 8 hours 1 hour
    P3 2 hours 24 hours 4 hours
    P4 8 hours 5 days 24 hours

    2. Automatic via monitoring systems
    3. Manual via user reports

    4. Documented in incident management system

    9. Investigation & Diagnosis

    10. Gather relevant logs and metrics
    11. Analyze system behavior

    12. Identify root cause

    13. Mitigation & Resolution

    14. Implement immediate fixes
    15. Restore service

    16. Verify resolution

    17. Post-Incident Review

    18. Document timeline
    19. Identify root cause
    20. Develop action items

    Disaster Recovery¶

    Recovery Scenarios¶

    Infrastructure Failure¶

    1. Single AZ Failure
    2. Automatic failover to other AZs
    3. No manual intervention required

    4. <5 minute recovery time objective (RTO)

    5. Region Failure

    6. Manual failover to backup region
    7. Data recovery from backups/replicas

    8. <1 hour RTO, <15 minute RPO

    9. Cloud Provider Failure

    10. Activation of disaster recovery plan
    11. Communication to all stakeholders
    12. <4 hour RTO, <1 hour RPO

    Data Corruption/Loss¶

    1. Database Corruption
    2. Point-in-time recovery from backups
    3. Data validation procedures

    4. <1 hour RTO, <5 minute RPO

    5. Accidental Data Deletion

    6. Restore from backups
    7. Validation of restored data
    8. <2 hour RTO, <24 hour RPO

    Recovery Procedures¶

    Database Recovery¶

    # Restore from snapshot
aws rds restore-db-instance-from-db-snapshot \
  --db-instance-identifier maily-prod-restored \
  --db-snapshot-identifier rds:maily-prod-2025-03-01 \
  --db-instance-class db.r5.xlarge \
  --no-publicly-accessible \
  --db-subnet-group-name maily-prod-subnet-group \
  --vpc-security-group-ids sg-12345

# Point-in-time recovery
aws rds restore-db-instance-to-point-in-time \
  --source-db-instance-identifier maily-prod \
  --target-db-instance-identifier maily-prod-restored \
  --restore-time 2025-03-01T12:00:00Z
